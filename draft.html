Why imp

Stop mistaking narration for completion.

Prompting is not a workflow. A long transcript is not memory. A confident paragraph is not proof. imp is what you use when you want the agent to actually touch the repo, run the checks, and leave behind durable state.

Terminal-native

Lives where the work already lives.

No detour into some toy sandbox. Shell, repo, tools, runtime. Right there.

Tool-using

Reads. Edits. Scans. Verifies.

It uses real tools by default instead of pretending reasoning alone is the product.

Verified

imp is built around evidence, targeted checks, and explicit closure instead of vibes.

Workflow

From task to verified result. No hand-waving.

The happy path is simple: work lives durably, imp pulls the right context, executes with tools, verifies the result, and writes the outcome back so future workers do not have to reconstruct reality from chat logs.

01

Units, dependencies, facts, verify gates, and prior evidence survive beyond any one session or model.

02

imp assembles the right context.

Not the entire transcript. The relevant repo state, files, prior evidence, and task contract.

03

It executes real work.

Use tools, edit code, run checks, respect policy, and keep the operator legible to what actually happened.

04

Results go back durable.

Future workers inherit progress, evidence, and closure instead of raw transcripts and wishful thinking.

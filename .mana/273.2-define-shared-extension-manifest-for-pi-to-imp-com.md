Goal: define the minimum extension package manifest/metadata contract imp needs to detect and stage existing Pi TypeScript extensions during import.

Grounding from local Pi audit:
- Pi extension root is `~/.pi/agent/extensions`.
- Shape is mixed: top-level `.ts` files and extension directories.
- Directory packages commonly use `package.json` with `pi.extensions: ["./index.ts"]` and optional local dependencies.
- Some package directories may lack `pi` metadata but still have `index.ts` and should be detected as fallback candidates.
- Runtime API pattern is default exported function receiving `ExtensionAPI`; extensions call `pi.registerTool`, `pi.registerCommand`, and `pi.on`.
- Tool schemas commonly use `@sinclair/typebox` values as `parameters`.

Minimum imp metadata to derive/store during import:
1. `name`: source filename stem or directory name, with optional package.json name retained as display/package name.
2. `source`: `pi`.
3. `runtime`: `typescript`.
4. `entrypoints`: from `package.json.pi.extensions`, single-file path, or fallback `index.ts`.
5. `package_dir` / copied path under imp config.
6. `status`: `staged` or `unsupported-runtime` until executable TypeScript runtime support lands.
7. Optional dependency marker: package has `package.json` dependencies/package-lock, useful for future runtime adapter.

Compatibility notes:
- Existing imp Lua extensions remain separate; this is import/staging metadata for Pi TS packages, not a Lua replacement.
- Static extraction of registered tools/commands/events is optional and should not block first import support; dynamic registration requires the TS runtime adapter later.
- Unsupported-runtime messaging is required so import does not imply execution.

Acceptance:
- A concrete shared manifest sketch exists and explicitly covers Lua backward compatibility, Pi TypeScript importability, capability declarations, and unsupported-runtime behavior.

  2026-04-15T03:14:52.488371+00:00
  Inspected auth resolution. Main findings:
  1. `crates/imp-core/src/imp_session.rs` special-cases `openai` in `resolve_api_key()` to `auth_store.resolve_api_key_only("openai")`, which bypasses saved OpenAI/ChatGPT OAuth even though `AuthStore::resolve("openai")` would prefer stored OAuth before api-key/env sources.
  2. In `ImpSession::create()`, when `options.no_tools` is true, auth resolution errors are swallowed via `resolve_api_key(...).await.unwrap_or_default()`. For headless `imp run` smoke tests this becomes an empty bearer token and matches the observed HTTP 401 `Incorrect API key provided: ''`.
  3. `AuthStore::set_runtime_key()` stores whatever string it receives; there is currently no trimming/empty-string guard. That is a real secondary footgun, but it was not required to explain the observed smoke-test failure because `cli.api_key` was unset.
  4. `AuthStore::resolve()` already implements the user's preferred precedence (`runtime override -> stored OAuth -> stored/api-key field -> env`). The bug is that `imp_session` / direct worker code bypasses that path for `openai`.
  5. There is also a consistency gap between CLI fallback logic (`should_use_chatgpt_provider` only auto-switches when `cli.provider` is unset) and session fallback logic (`should_use_codex` allows explicit `provider=openai` to fall back for codex-supported models). Worth reviewing after the primary auth fix, though not the direct cause of the nano smoke-test failure.

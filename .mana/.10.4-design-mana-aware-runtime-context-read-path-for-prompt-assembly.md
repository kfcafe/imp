## Goal
Define how imp runtime should read from the different memory layers without collapsing them together. Today the runtime already assembles a system prompt, injects frozen personal memory, uses task context, and supports headless file prefill for `imp run`, but there is no canonical read-path design for relevant mana facts / wiki pages in future flows.

## Current State
- `builder.rs` loads `memory.md` and `user.md` as frozen snapshots, passes `facts: &[]` (empty) to prompt assembly.
- `system_prompt.rs` has a Layer 4 (mana facts) slot, but it is never populated in `builder.rs`.
- `imp run <id>` in `main.rs` loads unit frontmatter, parses file paths, and injects context prefill via `context_prefill::assemble_context()`.
- Session compaction replaces older history with a summary boundary; `get_active_messages()` returns the compacted view.

## Scope
- In scope: design the read path for prompt assembly and headless dispatch.
- In scope: explain how session compaction boundaries interact with any mana/wiki-derived context.
- In scope: propose a Phase 1 integration path that preserves prefix caching and keeps runtime complexity bounded.
- Out of scope: fully implementing the runtime changes.

## Deliverable
- A proposal doc at `docs/proposals/mana-aware-runtime-context-read-path.md`.
- The doc should name specific call sites and recommend where future code changes belong.

## Patch Guidance
- Docs-only patch.
- Use inspected files and current behavior as the base reality.
- Prefer a phased plan over a maximal architecture that touches every turn immediately.

## Focus
- `crates/imp-core/src/builder.rs` (prompt assembly entry point; note `facts: &[]`)
- `crates/imp-core/src/system_prompt.rs` (Layer 4 facts slot)
- `crates/imp-core/src/imp_session.rs` and `session.rs` (active history and compaction)
- `crates/imp-cli/src/main.rs` and `context_prefill.rs` (headless `imp run` path, including unit `42.3` behavior)

## Required Questions to Answer
1. What should be loaded once at session start vs per-task / per-headless run?
2. Should wiki/fact context be injected as prompt text, file prefill, or on-demand query results?
3. How do we avoid stale or duplicate context alongside compaction summaries?
4. How do prefix caching and frozen personal-memory snapshots constrain the design?
5. What is the narrowest Phase 1 that adds value without destabilizing runtime?

Goal: determine whether the first public imp SDK should expose only `ImpSession` or also introduce a presentation-neutral runtime wrapper for host lifecycle.
Current state: `ImpSession` already handles core prompting/event flow, while CLI/TUI still own higher-level runtime/session replacement/orchestration concerns.

Decide and document:
1. Whether hosts need an `ImpRuntime`-like abstraction in SDK v1.
2. Which lifecycle responsibilities belong above `ImpSession` (session replacement, runtime recreation, extension/resource loading later, cwd-bound services, diagnostics).
3. Whether that wrapper is required now or should wait until after the first SDK surface is productized.
4. The clean boundary between session API and host/runtime API.

Files/context:
- `imp/crates/imp-core/src/imp_session.rs`
- `imp/crates/imp-cli/src/lib.rs`
- `imp/imp_rebuild_plan.md`
- `~/pi-mono/packages/coding-agent/docs/sdk.md` (benchmark only)

Deliverable:
- a short decision with rationale and follow-on implications.

Do not:
- copy pi mechanically
- overfit the answer to current TUI assumptions

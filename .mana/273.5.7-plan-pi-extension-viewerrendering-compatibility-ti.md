Goal: define how Pi TypeScript extension viewer/rendering functionality maps into imp so the first TS runtime sprint does not close the door on rich UI compatibility.

User concern:
- We may miss out on Pi viewer functionality when importing/executing TypeScript extensions.

Context from Pi examples:
- Official examples use tool-level `renderCall` and `renderResult` functions to customize display of tool invocations/results.
- `built-in-tool-renderer.ts` demonstrates renderer-only extensions overriding built-in tools for compact display.
- Some extensions use `ctx.ui.custom`, `ctx.ui.notify`, `ctx.ui.setStatus`, command UIs, and session/status widgets.

Design questions to resolve:
1. Which Pi viewer APIs exist and which current local/online fixtures use them?
2. What is the minimum compatibility tier for the sprint?
   - Tier 0: execute tools and ignore custom renderers with default text fallback.
   - Tier 1: preserve renderer metadata and expose a generic textual viewer result in imp.
   - Tier 2: map `renderCall`/`renderResult` into imp TUI rendering surfaces.
   - Tier 3: support custom interactive UI (`ctx.ui.custom`) and status widgets.
3. How should unsupported viewer APIs fail? Prefer clear fallback/warning over runtime crashes.
4. Can the adapter keep renderer callbacks registered for future use even if the first sprint does not render them?
5. Which fixture should prove viewer compatibility later? Candidate: official `built-in-tool-renderer.ts`; maybe too renderer-focused for first tool execution slice.

Recommended first-sprint stance:
- Do not implement full custom viewer/UI parity in the first execution slice.
- Do preserve `renderCall`/`renderResult` fields when registering TS tools if feasible.
- Default to imp's native tool rendering/text output when custom renderers are unsupported.
- Emit a clear unsupported-viewer warning in debug/status logs, not in every model-facing tool result.

Acceptance:
- There is a documented viewer compatibility tier plan and first-sprint fallback behavior, with at least one future fixture identified.

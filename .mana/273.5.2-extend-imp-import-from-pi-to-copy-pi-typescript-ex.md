Goal: extend the existing imp import command to import Pi TypeScript extensions alongside Pi skills and AGENTS.md.

Updated user direction:
- Extensions should come along naturally with Pi import because `imp import` already imports Pi skills.
- Avoid a separate extension-import mental model. If Pi is detected during import, Pi extensions should be detected/listed/imported alongside Pi skills.
- A `--from pi` filter can still be supported if it already exists, but users should not need to discover a special flag to get Pi extensions.

Current seam:
- CLI: `imp/crates/imp-cli/src/lib.rs::run_import`
- Core import detection/copying: `imp/crates/imp-core/src/import.rs`

Source shapes to support from `~/.pi/agent/extensions`:
- top-level `.ts` files
- directories with `package.json` containing `pi.extensions`
- fallback directories with `index.ts`

Destination:
- `.imp/extensions/pi/<name>/...` in the current project/workspace.

UX requirements:
1. Normal `imp import` detects/list/imports Pi TypeScript extensions when Pi is among detected sources.
2. `--dry-run` lists TypeScript extensions separately from skills and instruction files.
3. Confirmation prompt includes extension count.
4. Import result prints copied/skipped extension names and destination.
5. Already-existing names are skipped using the existing skip pattern.
6. Output clearly states executable/stubbed/unsupported status after import.

Tests:
- single-file extension import
- package directory import
- fallback index.ts directory import
- already-existing skip
- destination under `.imp/extensions/pi`
- normal import path includes extensions when Pi is detected

Out of scope:
- Remote Pi registry/import references.
- Full runtime parity.

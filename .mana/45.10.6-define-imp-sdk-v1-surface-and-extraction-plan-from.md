Goal: define the first public imp SDK shape so imp becomes a reusable hostable runtime, not only the best agent for working with mana.
Current state: imp-core already has partial programmatic surfaces (`ImpSession`, `SessionOptions`, `AgentBuilder`, `UserInterface`, `AgentEvent`), but they are not yet framed as a coherent SDK contract. The public surface is concentrated in imp-core internals, CLI/TUI still own host-specific orchestration logic, and there is no stable v1 story comparable to pi's SDK.

Inspect and decide:
1. What the current embeddable surface already is (`imp_core::imp_session`, `AgentEvent`, `UserInterface`, builder/runtime pieces).
2. What the SDK v1 should expose as first-class host APIs: session creation, runtime lifecycle, prompting, steer/follow-up, abort, event subscription/streaming, model/thinking changes, persistence control, UI bridge.
3. Whether SDK v1 should live in `imp-core` initially or behind a new crate/re-export surface.
4. Which host concerns remain out of scope for v1 (extension runtime, package loading, full TUI replacement, provider registration, viewer, etc.).
5. What refactors are required to make the SDK presentation-neutral across CLI/TUI/RPC/embedded hosts.

Files to inspect:
- `imp/crates/imp-core/src/imp_session.rs`
- `imp/crates/imp-core/src/agent.rs`
- `imp/crates/imp-core/src/builder.rs`
- `imp/crates/imp-core/src/ui.rs`
- `imp/crates/imp-core/src/lib.rs`
- `imp/crates/imp-cli/src/lib.rs`
- `imp/README.md`
- `imp/imp_rebuild_plan.md`
- compare against `~/pi-mono/packages/coding-agent/docs/sdk.md` for benchmark only

Deliverable:
- A compact SDK v1 design memo or mana note that names the intended public surface, the minimum host contract, the likely crate boundary, and the first extraction sequence.

In scope:
- SDK/embedding architecture
- public API shape
- extraction sequencing

Out of scope:
- full implementation
- TypeScript extension system design
- provider expansion

Do not:
- collapse imp into a thin mana-only client
- overfit the SDK around current TUI assumptions
- copy pi mechanically; preserve imp's stronger runtime/workflow stance

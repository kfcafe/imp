  Grounded redesign direction captured from the current conversation.

  Grounded critique from inspected implementation this session:
  - imp/crates/imp-tui/src/app.rs currently renders too many equal-priority surfaces (one-line top bar, optional widget tray, transcript area, optional sidebar, bottom editor, then many overlay modes), which makes the TUI feel like a capable terminal app rather than a focused cockpit.
  - app.rs remains a monolithic fullscreen controller mixing event loop, render composition, mode switching, sidebar logic, cached render orchestration, runtime signal handling, and key routing. This makes iterative cockpit-grade polish harder because new UI behavior naturally accretes as another mode/overlay instead of refining a coherent control grammar.
  - existing status surfaces are informative but not layered like an instrument cluster. The current top bar and status info do not yet cleanly separate identity, run state, attention, and deep detail.
  - the editor/composer is functional but still reads as a chat textbox more than a steering/control surface.
  - the sidebar is one of the strongest existing assets, but it still feels closer to a debug sidebar than a true telemetry rail.

  First-principles UX pillars for the next-gen TUI:
  1. Steering first — the operator should always know what Enter, Esc/cancel, queueing, and focus changes will do right now.
  2. Center = road, right rail = telemetry — transcript stays central; the sidebar evolves into a first-class subsystem/instrument rail.
  3. Two-tier top hierarchy — tier 1 identity (repo/session/elapsed), tier 2 run state (idle/thinking/acting/streaming/queued/blocked/review plus model, thinking, context pressure, active tools).
  4. Motion only proves liveness — subtle, low-amplitude indicators only for alive/waiting/active/focus states; no decorative animation or layout motion.
  5. Modes should feel like drive modes, not ad hoc overlays — Drive, Inspect, Review, Control Room, Browse are the target conceptual groupings.
  6. Tight canonical state vocabulary — idle, thinking, acting, streaming, queued, blocked, review, done, error.

  Phased decomposition implied by the conversation:
  - First, define a presentation-neutral cockpit state model and layout grammar so the TUI is rendered from deliberate identity/run-state/telemetry/composer/attention strata rather than ad hoc app.rs conditionals.
  - Second, map the app.rs/view seams needed to split layout/view composition from runtime/controller concerns, especially around sidebar -> telemetry rail and top-bar/status ownership.
  - Third, land the first proving slice in code: two-row instrument header, demoted/rethought widget tray, telemetry rail treatment, control-stick composer hints, and canonical state vocabulary.
  - Fourth, follow with first-class blocked/review cockpit states and drive-mode regrouping after the proving slice is stable.

  Recommended first proving slice A from this thread:
  - Replace the current single-line top bar with a two-row header.
  - Remove or demote the generic widget tray unless a widget truly deserves instrument-cluster prominence.
  - Recast the sidebar as telemetry with stronger active/selected/failure states.
  - Rewrite composer hints and footer state so the bottom feels like steering rather than a plain chat box.
  - Tighten visible state language everywhere to the canonical vocabulary above.

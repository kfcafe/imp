Implement guided CLI parity flows for the richest current TUI control surfaces once the first proving slice lands. Ground this work in the specs under 50.6.4 and the current code-backed capability inventory from `SettingsState`, `PersonalityState`, and `WelcomeState`. Scope:
1. add a real terminal-native `imp settings` / `:settings` flow with structured editing, typed prompts, and explicit save/apply semantics
2. add a real `imp personality` / `:personality` flow preserving builder vs source mode and global vs project scope
3. add a real `imp setup` / `:setup` onboarding flow preserving provider auth, model/thinking choice, and optional web-search setup
4. make these flows fullscreen-optional: they must work well in a plain terminal without requiring the TUI
5. permit optional richer handoff to TUI/editor surfaces where helpful, but keep CLI parity first-class

Relevant files likely include new CLI flow modules plus shared config/auth/personality actions, not just direct reuse of current TUI widgets.

Out of scope:
- generic browse/log/tree viewer work
- changing the underlying config/auth/personality storage model unless needed for adapter cleanup
- forcing users into fullscreen UI for edits

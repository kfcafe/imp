Goal: refactor direct headless `imp run` OpenAI runtime selection so the worker path is model-first, inherits defaults predictably, and hides transport/auth branching behind a cleaner runtime connection layer.

Current state:
- `openai` and `openai-codex` are currently modeled as peer provider identities across runtime selection, auth resolution, and tests.
- In practice they are two OpenAI-owned transport/auth surfaces with different endpoints and request contracts.
- This leaks implementation detail into user-facing runtime logic and makes direct `imp run` behavior confusing.
- The user expects direct `imp run` to inherit configured model/thinking defaults (e.g. `gpt-5.4` + `xhigh`) unless explicitly overridden.

Steps:
1. Inspect the current selection/auth flow across `crates/imp-cli/src/lib.rs`, `crates/imp-core/src/imp_session.rs`, and `crates/imp-llm/src/model.rs`.
2. Introduce a shared runtime-connection abstraction that accepts desired model/thinking/provider intent and returns the chosen route/surface plus credential requirements.
3. Refactor direct headless worker/session startup to use that abstraction instead of ad hoc `openai` vs `openai-codex` branching.
4. Preserve transport-specific provider implementations and request builders, but move their distinction below the shared runtime connection layer.
5. Make default inheritance explicit: CLI flags > unit assignment override > config defaults.
6. Add focused tests for default inheritance, route selection, and legible selection behavior.

Files:
- `crates/imp-cli/src/lib.rs`
- `crates/imp-core/src/imp_session.rs`
- `crates/imp-core/src/mana_worker.rs`
- `crates/imp-llm/src/model.rs`
- `crates/imp-llm/src/providers/openai.rs`
- `crates/imp-llm/src/providers/openai_codex.rs`

In scope:
- direct `imp run` runtime selection clarity
- OpenAI route/surface modeling
- default inheritance behavior

Out of scope:
- collapsing the two transport implementations into one if their wire/auth contracts still differ
- broader non-OpenAI provider redesign

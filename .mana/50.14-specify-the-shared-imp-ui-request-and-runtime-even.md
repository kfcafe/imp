Write a seam-spec doc defining the presentation-neutral event and UI-request contracts that should be shared across `imp chat`, `imp view`, RPC/machine surfaces, and `imp-tui`. Ground it in current inspected code and docs: `imp/crates/imp-tui/src/tui_interface.rs` already adapts `imp_core::ui::UserInterface` into `UiRequest`; `imp/crates/imp-tui/src/app.rs` consumes `AgentEvent` plus `UiRequest`; `imp/crates/imp-cli/src/main.rs` already hosts CLI-first shell and viewer flows. The spec should clarify which request/event categories must be shared (notifications, confirm/select/input prompts, status/widget updates, tool/runtime progress, session/turn events) and which behaviors remain presentation-specific. Explicitly connect this seam to the TUI adapter-boundary decision: the TUI should consume a shared contract instead of defining fullscreen-only interaction semantics. Keep the focus on ownership and contract shape, not implementation. Relevant files: `docs/rebuild/imp-tui-adapter-boundary.md`, `imp/crates/imp-tui/src/tui_interface.rs`, `imp/crates/imp-tui/src/app.rs`, `imp/crates/imp-cli/src/main.rs`, and shared imp-core UI/runtime types currently in use. Acceptance: produces a doc that names the shared contract categories, maps current `UiRequest`/`AgentEvent` grounding into the future shared seam, and states what remains TUI-local versus shared across surfaces.

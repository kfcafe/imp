## Task
Write `docs/rebuild/imp-attach-path-cutover.md` sequencing the cutover from the current `imp run <unit-id>` + legacy `mana run` coexistence into the long-term lease-based attach model where `imp run {mana_id}` attaches a live runtime/session to a mana-owned run.

## Current inspected state
- `docs/rebuild/mana-imp-ownership-boundary.md` already fixes the core ownership split: mana owns durable/shared/coordinated truth; imp owns live execution and final assembly.
- `docs/rebuild/runner-protocol-surface.md` already defines the first typed runner request/result groups for worktree, sandbox, command execution, capture, and cancellation.
- `docs/rebuild/imp-run-primary-execution-path.md` audits why `imp run <unit-id>` is already the strongest current primary path and why `mana run` remains compatibility behavior.
- `docs/rebuild/mana-lease-model.md` already defines execution/worktree/runner/sandbox lease vocabulary.
- `mana/crates/mana-core/src/ops/run.rs` currently exposes scheduling helpers (`ReadyQueue`, `RunPlan`) rather than attach/heartbeat/resolve semantics.
- `mana/crates/mana-cli/src/commands/run/mod.rs` still documents `mana run` around dispatch/spawn behavior.
- `imp/crates/imp-core/src/mana_worker.rs` already acts as the canonical single-unit worker runtime in current repo reality.

## Steps
1. Inspect the docs and code paths listed below.
2. Audit what parts of the eventual cutover already exist in repo reality versus what is still only implicit.
3. Sequence the staged migration captured in root mana notes on `45` into concrete phases, at minimum covering:
   - canonical run/node/lease/event/artifact/checkpoint semantics in mana
   - dual-write / compatibility recording through mana-owned semantics
   - shadow scheduling / shadow validation of legality and readiness
   - cutover to lease-based execution where imp attaches, heartbeats, checkpoints, and resolves through mana
   - transcript ownership fully in imp with mana as the structured recovery substrate
   - deprecation/narrowing plan for `mana run`
4. For each phase, name the likely owning project (`mana`, `imp`, or shared contract work), the key touched files or docs, and the narrowest proving slice.
5. Call out what can remain compatibility glue temporarily versus what must become canonical before later phases proceed.

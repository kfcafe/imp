This epic turns the current imp-core monolith into a staged decomposition plan. Goal: stop `imp-core` from remaining the place where every new responsibility lands.

Primary hot spots:
- `crates/imp-core/src/agent.rs`
- `crates/imp-core/src/tools/mod.rs`
- `crates/imp-core/src/session.rs`
- `crates/imp-core/src/tools/mana.rs`
- `crates/imp-core/src/config.rs`
- `crates/imp-core/src/mana_worker.rs`

Important decisions to resolve explicitly:
1. what should become new crates vs internal modules first?
2. what is the safest extraction order that preserves product behavior?
3. should `mana_worker` remain in imp or move behind the shared protocol layer first?
4. which hot spots are behavior-critical enough to refactor behind adapters?

Suggested child jobs:
- split `agent.rs` responsibilities into turn loop, tool execution coordination, event/timing emission, and follow-up logic
- split `tools/mod.rs` into tool contracts/registry vs support utilities
- separate session persistence from title/summary/replay heuristics
- carve worker-specific runtime out of `imp-core`
- identify the smallest reversible extraction seams for each hot-spot file

Done when:
- there is an execution-ready decomposition roadmap with concrete first slices and verify strategy

Goal: evaluate how karpathy/autoresearch patterns should be adapted into Tower to improve imp and mana without turning production runtimes into uncontrolled self-modifying systems.

Current state:
- imp already has early eval/benchmark surfaces: imp-core tool_ab_harness example, usage reporting, compaction logic, hooks, and native mana run/evaluate support.
- vNext phase 3 already calls for evaluation, regression harnesses, and observability.
- mana already provides durable task graph, verify gates, run/evaluate surfaces, facts, notes, and decisions that could store experiment lineage.
- autoresearch contributes a useful pattern: bounded experiment loop over a narrow mutable surface, fixed budget, objective metric, keep/discard policy, and human-reviewed experiment log.

Questions to resolve:
1. Which parts of autoresearch fit Tower directly: eval harnessing, experiment scheduling, result storage, prompt/program versioning, or selective code mutation loops?
2. What should remain out of scope initially: autonomous repo-wide self-editing, unreviewed merges, or production prompt mutation without guardrails?
3. How should imp and mana split responsibilities for experiment execution, durable evidence, and leaderboard/provenance?

Deliverables:
- recommended adaptation pattern for imp+mana
- minimal proving slice proposal
- risk/guardrail section
- mapping from autoresearch concepts to Tower surfaces

Files/areas to inspect:
- imp/vNext.md
- imp/crates/imp-core/examples/tool_ab_harness.rs
- imp/crates/imp-core/src/compaction.rs
- imp/crates/imp-core/src/hooks.rs

- docs/architecture/mana-platform-target-architecture.md

In scope:
- design and planning
- evaluation architecture
- durable experiment/evidence modeling
- sequencing for a safe first slice

Out of scope:
- landing autonomous self-editing loops in production
- broad plugin/runtime changes
- claiming that TS extensions or broader platform pieces already ship

Do not:
- propose uncontrolled self-modification of imp or mana
- bypass review/approval expectations
- blur mana platform ownership and imp product/runtime ownership

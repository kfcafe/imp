Goal: replace the current proof/spike Node bridge with a Bun-backed TypeScript extension runner for imp.

Context:
- A first TS adapter spike currently uses Node plus a temporary ESM loader to execute simple Pi-style TS extensions.
- User questioned this correctly: Pi extensions are `.ts` entrypoints and Bun is a better default runtime for direct TypeScript execution and package-local dependencies.
- Treat the Node bridge as a spike/proof, not the production adapter.

Scope:
1. Introduce a Bun-backed runner for TypeScript extension entrypoints.
2. Keep the Rust host as the authority for discovery, tool registration, policy, cancellation/timeouts, and result normalization.
3. Move Pi-specific shim/prelude behavior out of the generic runner where practical.
4. Add a clear actionable error when Bun is missing.
5. Preserve existing minimal tests or update them to skip/fail clearly depending on Bun availability.
6. Verify simple `registerTool` extension execution still works.

Out of scope:
- Full Pi API parity.
- Viewer/UI parity.
- Worker execution through TypeScript.

Acceptance:
- TypeScript extension execution uses Bun for `.ts` entrypoints.
- Node bridge is removed or explicitly demoted to fallback/spike-only code.
- Missing Bun produces a helpful user-facing error.
- Targeted TS extension tests pass where Bun is available.

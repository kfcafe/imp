Goal: Resolve the naming/product semantics around imp's `scan` tool as part of the broader Dirac-inspired code tooling work.

Context:
- User asked whether `scan` would be better named `search` or `codesearch`.
- `scan` currently does more than grep/search: it structurally scans directories/files, builds outlines, and extracts code blocks/symbols using tree-sitter.
- Dirac's comparable capabilities are named around intent: `get_file_skeleton`, `get_function`, `find_symbol_references`.
- User already clarified not to add extra model-facing tools for v1; focus is improving existing `read`, `scan`, and `edit`.

Tasks:
1. In `docs/design/dirac-inspired-code-tools.md`, include a naming subsection comparing:
   - `scan`
   - `search`
   - `codesearch`
   - possible action-level labels like `scan` action `extract`, `build`, `search`, `skeleton`.
2. Evaluate each option against:
   - model discoverability.
   - user clarity.
   - semantic accuracy.
   - collisions with text search/web search.
   - backwards compatibility and prompt/tool-call breakage.
   - future ability to support symbol references/semantic search.
3. Recommend a v1 decision. Suggested default unless evidence says otherwise:
   - keep canonical tool name `scan` for compatibility.
   - rewrite description to say it is for code structure search, outlines, and symbol/block extraction.
   - improve action names/descriptions/examples so models reach for it before shell+rg or full file reads.
   - consider optional alias `code_search` only after benchmark evidence shows better model behavior.
4. Define a decision rule/eval: compare tool-call choice on a small prompt set before/after descriptions/aliases.

Out of scope:
- Implementing rename/alias.
- Adding new tools in v1.

Do not:
- Recommend a breaking rename without a compatibility layer.

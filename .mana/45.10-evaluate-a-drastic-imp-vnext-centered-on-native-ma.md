Goal: capture the design space for a more radical imp rework/fork so future planning can compare it against the current in-place rebuild strategy.
Current state: Tower already frames imp as the native worker/runtime designed to use mana well, with active rebuild work around clearer worker/runtime seams, contracts, and runner boundaries. The open question is what a more drastic vNext would look like if mana became a first-class native task/memory substrate inside imp, extensibility moved beyond Lua, and the runtime became more portable/hostable.

Explore and document:
1. What should stay in imp core versus move into a portable host/runtime seam.
2. What it means for mana to be a native task+memory tool rather than an adjacent integration.
3. Candidate extension models beyond Lua (guest runtimes, capability-scoped tools, declarative modules, WASM/plugin boundary, RPC adapters) and their tradeoffs.
4. Portability requirements: terminal app, headless worker, embedded library, background agent, remote/isolated runner.
5. Whether the recommended path is still in-place refactor, a shadow vNext inside the repo, or a true fork.
6. Migration constraints: preserve mana↔imp contract, avoid duplicated durable truth, keep runtime policy enforceable by the Rust host.

Goal: make human-facing streamed errors consistent and user-friendly across imp-cli and imp-tui by moving provider/API error normalization into shared imp-core code.

Current state:
- imp-tui/src/app.rs has local parse_api_error()/HTML-collapse helpers.
- imp-cli/src/lib.rs prints raw AgentEvent::Error strings in both print and interactive/headless paths.
- This duplicates behavior and leaves CLI with much noisier provider/auth/gateway failures.

Steps:
1. Add a shared imp-core error-normalization helper that can accept raw provider/stream error strings and return a concise display string while preserving the original raw string for machine-facing use.
2. Move or duplicate the current TUI JSON/HTML/auth parsing behavior into that shared helper, including expired-token guidance.
3. Update imp-tui to use the shared helper instead of its local copy.
4. Update imp-cli human-facing paths to use the same helper for AgentEvent::Error and tool-error summaries where appropriate.
5. Add focused tests covering nested JSON provider errors, HTML error pages, plain-text passthrough, and auth-expired hints.

Files:
- imp/crates/imp-core/src/... (new helper module)
- imp/crates/imp-cli/src/lib.rs
- imp/crates/imp-tui/src/app.rs

In scope:
- Human-facing rendering consistency
- Reuse between CLI and TUI
- Focused unit tests

Out of scope:
- Changing machine-facing JSON/RPC event schema
- Provider retry policy changes unless required to support clearer messages

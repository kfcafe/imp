Goal: turn current capability concerns into a concrete policy design. Current state: mode and tool restrictions exist, but deep review shows gaps between intended boundaries and actual runtime-enforced behavior, especially for Lua/guest runtime access and cancellation.

In scope:
- define the deterministic runtime policy interface
- identify deny-by-default capability assumptions for guest runtimes
- decide which checks belong at prompt time vs execution time
- define how cancellation, env access, shell access, and network access should be enforced consistently

Out of scope:
- implementing every policy hook immediately
- expanding extension features

Important decisions to resolve explicitly:
1. what are the core policy questions the runtime must answer?
2. should Lua remain a first-class extension identity or become one backend of a stricter guest-runtime abstraction?
3. what are the default capabilities for guest runtimes?
4. where should policy decisions be logged and surfaced for debugging/review?

Suggested follow-ups:
- add runtime policy interface and decision types
- tighten Lua/guest runtime access to native tools, env, shell, and network
- align mode-visible tool lists with actual executable capability
- propagate cancellation through active tool execution

Goal: define the smallest robust provider failure taxonomy and retry/fallback policy for imp.

Current state:
- imp has a generic provider abstraction and retry helper, but lacks a clearly documented taxonomy for auth failures, rate limits, transient server errors, context overflow, payment/quota failures, and model incompatibility.
- Hermes has much richer classification logic (`agent/error_classifier.py`) and uses it to steer fallback behavior.

Steps:
1. Inspect current imp surfaces relevant to provider failure behavior:
   - `/Users/asher/imp/crates/imp-llm/src/provider.rs`
   - `/Users/asher/imp/crates/imp-core/src/retry.rs`
   - provider implementations under `/Users/asher/imp/crates/imp-llm/src/providers/`
2. Produce a concise design note or mana-backed recommendation that classifies the main error buckets imp should recognize.
3. For each bucket, specify:
   - retryable vs non-retryable
   - whether model fallback or provider fallback is appropriate
   - whether the user should be prompted or given an auth/config hint
4. Keep the proposal implementation-oriented and sized for the current Rust architecture.

In scope:
- failure taxonomy
- retry/fallback policy
- user-facing auth/config guidance rules

Out of scope:
- large provider refactors in this unit
- building a full credential pool

Do not:
- copy Hermes error handling wholesale
- propose hidden magic fallback behavior that obscures failure causes

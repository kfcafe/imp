## Task
Specify what a parent imp receives back from a delegated child imp call in the first delegation slice.

## Must answer
- Does the parent receive final text only, or structured outcome objects?
- Are stream/progress events surfaced synchronously, asynchronously, or summarized?
- How are verify status, logs, session IDs, unit IDs, and artifact references returned?
- What minimal fields are required so the parent can reason without scraping raw text?

## Constraints
- Reuse existing repo concepts where possible: `WorkerAssignment`, `WorkerResult`, `WorkerStatus`, run views/events, mana logs, and unit IDs.
- Avoid inventing a second incompatible result vocabulary if current contracts or Tower-level contracts can absorb the shape.
- Distinguish transient child output from durable mana-recorded outcomes.

## Acceptance
- Parent-facing result contract is explicit enough to implement a first native `imp` tool.
- The design identifies which fields are local runtime results versus durable references into mana/log/session state.

## Verify
Parent design note includes a section naming the parent-visible result contract and references structured outcomes or result fields.

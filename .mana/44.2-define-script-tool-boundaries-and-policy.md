Goal: define the script tool as a distinct product/runtime surface from durable extensions.
Current state: there is no dedicated script tool. Discussion so far indicates that one-off sandbox scripting should probably appear to the agent as a tool, while durable extensions should register ordinary tools via the extension substrate.

Steps:
1. Write a focused design note in `docs/proposals/` covering the script tool only.
2. Define when the agent should see script execution as an explicit tool versus ordinary extension-provided tools.
3. Specify the tool contract: inputs/outputs, runtime selector, timeout/memory/resource policy, capability model, and whether it can access native tools directly.
4. Define product boundaries between `extend` (authoring/management), durable extensions (registered tools/hooks/commands), and ephemeral script execution.
5. Capture open questions that should remain unresolved until after the substrate proposal lands.

Files:
- `docs/proposals/script-tool-boundaries-and-policy.md` (create — focused script tool proposal)
- `crates/imp-core/src/tools/extend.rs` (read — current authoring tool scope)
- `crates/imp-lua/src/bridge.rs` (read — current host callable surface)
- `docs/proposals/guest-runtime-extension-substrate.md` (read after created — align terms and boundaries)

Context:
- Script execution is for one-off transforms/mini-programs during a run.
- Durable extension tools should usually hide runtime details from the agent; the agent mostly sees normal tool names.
- The proposal should keep the script tool policy-first and avoid committing to a specific JS engine now.
- It should explicitly explain why the script tool is not the same thing as improving `extend`.

In scope:
- product boundary, safety policy, agent-facing UX, API sketch

Out of scope:
- implementation
- specific JS engine choice
- extension marketplace/installation UX

Do not:
- merge script execution and durable extension management into one concept
- assume raw filesystem/network/process access
- define a tool that bypasses Rust-side policy enforcement

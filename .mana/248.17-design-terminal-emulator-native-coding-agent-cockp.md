Goal: externalize a concrete design/decomposition for integrating a coding-agent operator surface directly into a terminal emulator, grounded in the current Tower direction and usable by another worker cold.

Current state:
- Existing root work already covers CLI-first shell design (`50.6`), simplified fullscreen TUI role (`50.8`), and a cockpit-style TUI redesign direction (`248.16`), but the conversation added a more specific terminal-emulator-native framing that spans shell awareness, pane/worktree isolation, approval/trust surfaces, diff-first editing, and resumable task state.
- The current discussion argues that terminal integration should not just bolt chat onto a shell. The terminal host should understand execution, provenance, verification, isolation, and operator control.
- This is primarily an imp operator-surface/design problem, but some affordances touch the mana/imp boundary because durable handoff, task state, evidence capture, and resumability matter.

Steps:
1. Synthesize the conversation into a durable design note or mana-backed spec that defines the terminal-emulator-native operator model.
2. Organize the design into explicit capability buckets:
   - structured shell awareness
   - inline command mediation / approval policy
   - diff-first editing and review flow
   - pane-native collaboration and multi-agent layout
   - diagnostics/test feedback parsing
   - queryable scrollback / evidence timeline
   - repo-aware context assembly
   - worktree/sandbox isolation primitives
   - agent observability and provenance
   - durable handoff / resumable task state
3. Separate MVP from later slices. The MVP from the conversation is: structured shell awareness, diff-preview editing, risky-command approval gates, test/error parsing, pane/worktree isolation, and session bookmarks/resumable task state.
4. For each capability bucket, classify likely ownership:
   - terminal host / emulator integration
   - imp runtime/operator surface
   - mana durable substrate
5. Explicitly connect this design back to existing root units `50.6`, `50.8`, `248.16`, and any isolation/policy tracks that should inherit the framing.

In scope:
- design/decomposition for a terminal-emulator-native coding-agent surface
- ownership split between terminal host, imp, and mana
- operator trust/approval/isolation model
- phased MVP sequencing

Out of scope:
- implementing a terminal emulator
- committing to a specific external terminal product or plugin architecture
- replacing the existing CLI-first/TUI role work; this unit should refine and connect it

Do not:
- reduce the idea to a chat sidebar in a terminal
- assume shell access alone is sufficient without structured provenance and approval
- reintroduce IDE-style chrome when a terminal-native model is stronger

Goal: diagnose and fix the residual worker-loop termination bug where fresh successful one-turn headless worker runs still end with `Max turns exceeded (1)` after already emitting the correct response and stop assessment.

Current state:
- Fresh cargo-built smokes for both `openai-codex/gpt-5.3-codex` and plain `openai/gpt-5.4` now stream `SMOKE_OK` successfully.
- After `message_end` and a `turn_assessment` whose chosen action is `stop` with reason `no_automatic_follow_up`, the process still emits `{"error":"Max turns exceeded (1)"}`.
- This indicates the original auth/instructions regressions are fixed, but the worker runtime still mishandles clean termination at the turn budget boundary.

Steps:
1. Inspect the headless worker run loop / max-turn handling in `crates/imp-core` to find why a successful stop assessment still trips the max-turn error path.
2. Add a focused regression test for a one-turn successful headless/no-tools worker run that should exit cleanly without `Max turns exceeded`.
3. Fix the termination logic and verify the smoke path exits successfully after the correct response.

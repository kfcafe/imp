## Goal
Define the implementation-facing execution contract for write-oriented semantic actions in `imp` after the typed policy model exists.

## Current state
- `docs/design/imp-write-oriented-code-intelligence-policy.md` captures the architecture and policy boundary.
- `docs/design/imp-semantic-write-policy-model.md` now captures the typed policy/config model, capability matrix shape, posture enums, escalation inputs, and preview/result envelopes.
- What still remains implicit is the exact runtime sequencing and state transitions around preview generation, approval gating, checkpoint timing, hosted apply, freshness revalidation, verify execution, and restore behavior.

## Scope
Planning only. No implementation.

## Deliverable
Create `docs/design/imp-semantic-write-execution-contract.md` as a planning artifact that another worker can implement cold.

## Required structure
1. A step-by-step execution flow for semantic writes from request -> capability lookup -> preview -> approval -> checkpoint -> apply -> semantic refresh -> verify -> result/restore surface.
2. Explicit rules for when preview is recomputed or rejected because workspace/backend state is stale.
3. Explicit checkpoint timing and labeling rules, including what is captured and what restore affordance must remain visible.
4. Approval handshake rules tied to the typed `ApprovalPosture` model, including what counts as operator-visible preview acknowledgement.
5. Apply-path rules that keep semantic writes on a hosted edit path inside the workspace boundary rather than delegating arbitrary backend commands.
6. Failure handling for: preview too large, apply drift, partial apply, semantic refresh failure, verify failure, approval rejection, and operator cancellation.
7. Result/receipt rules that consume the `SemanticWriteResult` envelope from the typed policy model and clarify what is durable evidence vs live runtime output.
8. Worked examples for rename, organize imports, and an allowlisted code action.

## Constraints
- Keep this specific to write-oriented semantic actions.
- Do not broaden into generic extension/sandbox policy.
- Do not collapse semantic writes into normal `edit`/`write` flows.
- Treat backend success as secondary evidence; explicit verify remains mandatory.

- Preview, checkpoint, approval, and verify are each modeled as distinct stages.
- Restore/rollback visibility and failure paths are specified.
- The note stays specific to semantic-write runtime behavior.

Current state:
- `51.1` now wires bounded mana facts into `imp` session-start prompt assembly via `mana_core::api::find_mana_dir()` and `mana_core::api::memory_context()`.
- Interactive startup still does not inject the non-fact portions of mana memory context: `warnings`, `working_on`, and `recent_work`.
- We intentionally deferred that in `51.1` because Layer 4 already cleanly owned facts, while status-like memory context needs its own prompt seam instead of being mixed into the fact layer.

Task:
Add the next minimal prompt-assembly seam for interactive `imp` startup so compact project-memory status can be injected separately from facts when a `.mana/` dir is available.

Concrete requirements:
1. Inspect `imp/crates/imp-core/src/system_prompt.rs` and choose a clearly named optional layer/field for compact project memory status, distinct from Layer 4 facts.
2. In `imp/crates/imp-core/src/mana_prompt_context.rs` (or a nearby helper), add bounded formatting for a compact status block derived from `mana_core::api::memory_context()` covering only the smallest useful subset of:
   - warnings
   - working-on
   - recent work
3. Keep the block selective and bounded. Do not dump raw unit bodies or large notes. Prefer a small preview-oriented format suitable for session-start context.
4. Preserve resilience: missing `.mana/` or load/format failures must degrade to no-op context.
5. Update `builder.rs` to pass the new optional status block into prompt assembly alongside the already-wired facts.
6. Add focused tests around formatting/plumbing in `imp-core`; avoid broad workspace checks.

Relevant files:
- `imp/crates/imp-core/src/system_prompt.rs` (modify — add new optional prompt seam)
- `imp/crates/imp-core/src/builder.rs` (modify — pass status block)
- `imp/crates/imp-core/src/mana_prompt_context.rs` (modify — bounded status formatting)
- `mana/crates/mana-core/src/ops/memory_context.rs` (read)
- `mana/crates/mana-core/src/api/mod.rs` (read unless a new public seam is needed)

Acceptance:
Interactive `imp` startup should have a distinct, bounded path to inject compact project-memory status context (`warnings` / `working_on` / `recent_work`) without conflating it with facts and without making startup brittle when no `.mana/` exists.

Do not:
- Do not shell out to `mana context`.
- Do not duplicate already-injected facts in the new status block.
- Do not mix dynamic work-status context into the existing fact layer.
- Do not make `.mana/` presence mandatory for startup.

Goal: draft the first concrete shared contracts surface between mana and imp directly in mana so implementation can start from a stable typed boundary instead of ad hoc runtime structs and prose-only doctrine.

Current state:
- `imp_rebuild_strategy.md` and `../rebuild_plan.md` both recommend a protocol-first boundary.
- `crates/imp-core/src/mana_worker.rs` currently contains important worker-facing types that are cross-boundary in spirit but owned by imp.
- `mana-core` already owns durable unit, verify-history, dependency, and artifact-adjacent state, but the exact shared protocol layer is still implicit.
- The highest-value first move is not broad implementation. It is a clean, minimal v1 contracts draft that both sides can target.

In scope:
- define the first shared type families for a contracts crate
- recommend where the crate should live
- define ownership rules for each type family
- define what is in v1 vs deliberately deferred
- define the minimum provenance/artifact fields required for trustworthy handoff

Out of scope:
- implementing the crate fully in code
- migrating all existing runtime and substrate types immediately
- designing every future workflow artifact up front

## Recommended crate placement

Recommended default: a Tower-level shared crate, conceptually sibling to mana and imp, even if the first practical implementation lands under `mana/` or the root workspace for convenience.

Why:
- the protocol is shared and should not conceptually belong only to one side
- it reduces accidental ownership drift
- it makes future runner/reviewer/control-plane consumers easier to add

Acceptable first implementation compromise if workspace friction is high:
- place the crate where it is easiest to wire immediately, but document that the API is a Tower boundary and not a mana-internal or imp-internal detail

Important decision still to resolve explicitly before implementation:
1. Tower-root crate now vs temporary mana-owned crate with planned extraction later

## Ownership rules

### mana should own durable truth about
- units and dependency graph
- acceptance criteria and required verifiers
- verify history and durable verifier artifacts
- approvals and review decisions
- durable artifact references
- leases or records another worker must inherit cold

### imp should own live runtime state about
- active turn loop
- current context packing
- live tool execution
- local file caches and checkpoints
- in-session heuristics and adaptation
- UI/session-local rendering state

### shared contracts should own
- assignment payloads
- workflow-stage outputs
- evidence bundle shapes
- review/approval payloads
- worktree/sandbox/runner lease and result shapes when they cross boundaries

Practical rule:
- if another worker, reviewer, scheduler, or substrate process must trust it cold, the boundary type should exist in shared contracts and the durable record should live in mana

## Proposed v1 contract families

### 1. Task and planning contracts

These define what work is being asked for before execution starts.

Suggested first types:
- `TaskSpec`
- `AcceptanceCriterion`
- `VerifierRequirement`
- `TaskRisk`
- `TaskScope`
- `PlanStep`
- `ExecutionPlan`

Suggested shape:

```text
TaskSpec
- task_id
- title
- description
- acceptance_criteria[]
- non_goals[]
- risk
- scope
- required_verifiers[]
- paths[]
- dependencies[]
- model_hint?
```

Key design decisions to resolve:
1. Should acceptance criteria be plain strings in v1 or structured tagged records?
2. Should `ExecutionPlan` be in v1 or deferred until after assignment/result contracts are stable?
3. How much file/symbol targeting belongs in `TaskScope` now?

Recommendation:
- keep v1 simple but typed: structured records with string payloads, not just raw prose blobs

### 2. Worker assignment contracts

These define what mana hands to imp.

Suggested first types:
- `WorkerAssignment`
- `AssignmentContextRef`
- `AttemptSummary`
- `DependencySummary`

Suggested shape:

```text
WorkerAssignment
- assignment_id
- unit_id
- title
- description
- acceptance?
- verify_command?
- notes?
- decisions[]
- dependencies[]
- dependency_summaries[]
- paths[]
- files[]
- prior_attempts[]
- workspace_root
- model_override?
- required_verifiers[]
- risk
```

Mapping note:
- this should be derived initially from the current `imp-core/src/mana_worker.rs` assignment shape, but promoted into the shared boundary crate

Key decisions to resolve:
1. Is `workspace_root` part of the shared contract or local runner context?
2. Should dependency summaries be embedded or referenced by artifact ID in v1?
3. Which fields are mandatory vs best-effort?

Recommendation:
- embed enough for a cold worker to start without further graph lookups, but keep large evidence bodies referenced rather than duplicated

### 3. Worker outcome contracts

These define what imp reports back after a run.

Suggested first types:
- `WorkerOutcome`
- `StageStatus`
- `RunMetrics`

Suggested shape:

```text
WorkerOutcome
- run_id
- unit_id
- stage
- status
- summary?
- error?
- metrics?
- artifact_refs[]
- produced_review_needed?
- completion_candidate
```

Suggested statuses for v1:
- `completed_candidate`
- `awaiting_verify`
- `verified_complete`
- `complete_with_gaps`
- `needs_review`
- `blocked`
- `failed`
- `cancelled`

Important decision:
1. Which of these statuses are shared contracts vs mana-internal state machine labels?

Recommendation:
- keep the shared contract slightly richer than the current mana status enum, then let mana map them into durable unit state intentionally

### 4. Verifier and evidence contracts

This is the most important family after assignment/outcome.

Suggested first types:
- `VerifierResult`
- `EvidenceBundle`
- `ArtifactRef`
- `ArtifactKind`
- `ArtifactProvenance`
- `ScopeCheckResult`

Suggested minimum `VerifierResult` fields:

```text
VerifierResult
- verifier_name
- status (passed | failed | skipped)
- command?
- exit_code?
- summary?
- artifact_refs[]
- started_at
- finished_at?
- run_id
```

Suggested minimum `EvidenceBundle` fields:

```text
EvidenceBundle
- bundle_id
- unit_id
- run_id
- patch_ref?
- verifier_results[]
- scope_check?
- notes?
- completion_recommendation
```

Suggested minimum `ArtifactRef` fields:

```text
ArtifactRef
- artifact_id
- kind
- uri_or_path
- created_at
- provenance
```

Suggested minimum `ArtifactProvenance` fields:

```text
ArtifactProvenance
- unit_id
- run_id
- stage
- agent_id?
- worktree_id?
- sandbox_id?
- source_tool?
```

Important decisions to resolve:
1. What is the smallest mandatory evidence bundle for a coding run?
2. Do patch diffs get first-class artifact types in v1?
3. Which artifact references must be durable in mana vs allowed to remain ephemeral implementation detail?

Recommendation:
- require enough provenance to bind artifacts to a run/stage now; richer storage/backends can come later

### 5. Review and approval contracts

Suggested first types:
- `ReviewDecision`
- `ReviewFinding`
- `ApprovalDecision`
- `EscalationReason`

Suggested shape:

```text
ReviewDecision
- review_id
- unit_id
- run_id
- decision (pass | needs_changes | escalate)
- findings[]
- artifact_refs[]
- reviewer_kind (agent | human | rule)
```

```text
ApprovalDecision
- approval_id
- unit_id
- run_id
- approved (bool)
- reason?
- approver_kind
- artifact_refs[]
```

Important decisions:
1. Should review/approval contracts be in v1 or introduced only after evidence bundle types stabilize?
2. Should human and machine approvals share the same shape?

Recommendation:
- include simple shared review/approval shapes in v1 even if first implementations are sparse; it keeps the workflow model honest

### 6. Runner / environment contracts (v1-light)

Suggested first types:
- `WorktreeLease`
- `SandboxSpec`
- `RunnerRequest`
- `RunnerResult`

Recommendation:
- define them now, but keep them intentionally minimal until the runner track is ready

Possible v1-light shape:

```text
WorktreeLease
- worktree_id
- unit_id
- branch
- base_commit
- path
- owner_run_id
- created_at
```

```text
SandboxSpec
- sandbox_id
- profile (analysis | edit | verify | hostile)
- network_mode
- writable_paths[]
- secret_policy
```

```text
RunnerRequest
- run_id
- command
- cwd
- worktree_id?
- sandbox_spec?
```

```text
RunnerResult
- run_id
- status
- exit_code?
- artifact_refs[]
- started_at
- finished_at?
```

Important decisions:
1. Do these belong in the first contracts implementation or only as placeholder types?
2. Which fields must be durable in mana immediately?

Recommendation:
- include placeholder/shared shapes now so later runner work does not invent a parallel protocol surface

## Proposed v1 package surface

Suggested first public modules:
- `task`
- `assignment`
- `outcome`
- `evidence`
- `review`
- `runner` (lightweight for now)
- `ids` or `common`

Suggested common enums/types:
- `TaskRisk`
- `StageName`
- `CompletionRecommendation`
- `VerifierStatus`
- `ArtifactKind`

## What to defer

Do not overdesign these in v1:
- full graph query protocol
n- remote runner / daemon protocol details
- full TUI event contracts
- every future memory or retrieval artifact
- rich symbolic planning schema beyond a simple `ExecutionPlan`

## Suggested implementation sequence after this draft

1. create the contracts crate
2. land core IDs/enums/common types
3. land `WorkerAssignment`, `WorkerOutcome`, `VerifierResult`, `EvidenceBundle`, `ArtifactRef`
4. add compatibility adapters from current `imp-core/src/mana_worker.rs`
5. teach mana and imp to consume the shared types without changing outer behavior first
6. only then start deeper imp-core decomposition and evidence persistence work

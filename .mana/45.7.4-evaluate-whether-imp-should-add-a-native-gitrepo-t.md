Goal: decide whether imp should grow a native git-facing capability and, if so, what the minimum safe shape should be.

Current state:
- imp can already perform git operations through bash.
- The architectural question is not whether git commands are possible today, but whether a native tool would materially improve safety, determinism, isolation, provenance, and mana-integrated execution.
- The strongest candidate shape discussed so far is not a generic `git args...` passthrough, but a narrow, policy-aware repo/worktree surface.

Evaluate and capture:
1. The real advantages of a native git/repo tool over bash for imp, with emphasis on Tower-specific concerns: structured output, policy enforcement, multi-agent isolation, worktree handling, durable provenance, and consistent runtime boundaries.
2. Cases where a native tool is not worth the scope, especially if it degenerates into a generic shell-like git wrapper.
3. A recommended v1 boundary for the tool if it exists at all. Prefer a narrow repo/worktree API such as status/diff/log/branch/worktree/stage/commit/restore over full git coverage.
4. Which responsibilities belong in imp runtime policy vs mana durable state vs bash fallback. Be explicit about where worktree isolation metadata and evidence should live.
5. Risks, edge cases, and non-goals: force-push/rebase/reset-hard/destructive repo mutation, arbitrary git config mutation, and the danger of re-implementing all of git.
6. A recommendation on whether to proceed now, defer, or require another prerequisite.

Deliverable:
- Durable architecture note or equivalent mana-recorded conclusion that another worker can inherit cold.
- If the answer is yes, include a small candidate operation set and guardrail model.

Suggested files to inspect when doing the evaluation:
- imp/crates/imp-core/src/tools
- imp/crates/imp-core/src/agent.rs
- imp/crates/imp-core/src/mana_worker.rs
- mana/crates/mana-core and mana/crates/mana-cli surfaces relevant to durable execution evidence and worktree/lease ownership
- docs/rebuild/* and docs/architecture/* touching runner/policy/isolation if relevant

In scope:
- tool-boundary design
- policy and isolation implications
- mana↔imp contract implications for repo/worktree evidence

Out of scope:
- implementing the tool
- full git abstraction design
- changing remote/VCS workflows unless needed to explain a risk

Do not:
- propose a generic unrestricted `git(args...)` wrapper as the primary answer
- treat prompt instructions as sufficient enforcement when host/runtime policy is the real issue
- assume concurrent workers are safe without explicit isolation/worktree handling

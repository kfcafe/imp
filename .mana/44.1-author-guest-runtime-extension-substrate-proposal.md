Goal: author the main design proposal for a guest-runtime extension substrate in imp.
Current state: imp has a runtime-specific Lua path (`lua_tool_loader` in `crates/imp-core/src/builder.rs`, `crates/imp-lua/src/*`), and the current `extend` tool is focused on skills/Lua reference rather than general extension management.

Steps:
1. Write a proposal doc in `docs/proposals/` that defines the architectural shift from "Lua extensions" to "guest runtimes".
2. Specify core concepts: Rust host ownership, guest runtime interface, extension manifest, permissions/capabilities, durable extension lifecycle, and migration path from current Lua support.
3. Ground the proposal in inspected files: `crates/imp-core/src/builder.rs`, `crates/imp-core/src/tools/extend.rs`, `crates/imp-lua/src/lib.rs`, `crates/imp-lua/src/loader.rs`, `crates/imp-lua/src/sandbox.rs`, `crates/imp-lua/src/bridge.rs`.
4. Include explicit boundaries: what remains in Rust, what moves behind guest runtimes, and what not to do yet.
5. End with a phased implementation plan that future workers can decompose into code changes.

Files:
- `docs/proposals/guest-runtime-extension-substrate.md` (create — primary design note)
- `crates/imp-core/src/builder.rs` (read — current runtime-specific hook)
- `crates/imp-core/src/tools/extend.rs` (read — current authoring scope)
- `crates/imp-lua/src/lib.rs` (read — existing extension bootstrap)
- `crates/imp-lua/src/loader.rs` (read — extension discovery/loading)
- `crates/imp-lua/src/sandbox.rs` (read — runtime state and policy surface)
- `crates/imp-lua/src/bridge.rs` (read — host API)
- `docs/proposals/imp-memory-architecture-and-mana-boundary.md` (read — proposal style precedent)
- `docs/proposals/mana-aware-runtime-context-read-path.md` (read — proposal style precedent)

Context:
- The proposal should preserve the working Lua extension path as the first guest runtime rather than replacing it.
- JS/TS should be treated as a future possible runtime, not a required part of phase 1.
- The key architectural move is language-agnostic host capabilities and policy enforcement in Rust.
- The design should explain why this is bigger than a new tool and how it changes the extension substrate.

In scope:
- architecture, terminology, boundaries, migration framing, implementation phases

Out of scope:
- code implementation
- runtime benchmarking
- choosing a specific JS engine now

Do not:
- frame the design as "replace Lua with TS"
- overload the proposal with speculative engine details
- leave worker-facing steps vague; the phases should be concrete

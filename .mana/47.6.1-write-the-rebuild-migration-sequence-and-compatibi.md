Goal: turn the rebuild into a landable migration sequence. Current state: the strategy names the work tracks, but not yet the exact order of implementation slices, temporary adapters, and rollback points.

In scope:
- define the migration order across contracts, imp-core decomposition, runner boundary, evidence model, and policy hardening
- define which temporary re-exports or compatibility shims are acceptable
- specify verify commands and rollback points for the highest-risk slices
- call out which user-visible behavior must remain stable during the migration

Out of scope:
- executing the full migration
- broad product redesign while the runtime boundary is still moving

Important decisions to resolve explicitly:
1. which slices must block others because of ownership or protocol dependencies?
2. where should temporary adapters live, and what should force their removal later?
3. which risky slices need extra rollback preparation?
4. what parts of TUI/CLI should intentionally not move until lower layers are stable?

Suggested follow-ups:
- land contracts crate first
- then land the first imp-core extraction seam behind adapters
- then land runner protocol and local runner MVP
- then land evidence/completion model and policy hardening slices

Files to read:
- `imp_rebuild_strategy.md`
- `../rebuild_plan.md`
- current imp crate layout and hot-spot files
- current mana/imp boundary docs

Done when:
- the rebuild has a realistic migration order, compatibility approach, and rollback-aware verify plan

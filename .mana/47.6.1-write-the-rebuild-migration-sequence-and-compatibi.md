  Concrete migration sequence after inspecting imp_rebuild_plan.md, imp_rebuild_strategy.md, and current hot spots:

  Recommended landable order
  1. Stabilization / freeze phase
     - Treat current large files as controlled hot spots: imp-core/src/agent.rs (~5700 LOC), session.rs (~2300), tools/mod.rs (~1200), tools/mana.rs (~3500), config.rs (~1400), mana_worker.rs (~1000), and imp-tui/src/app.rs (~6800).
     - Do not begin broad TUI cleanup while lower runtime seams are moving.
     - Preserve current top-level operator surfaces during migration: `imp` CLI entry, `imp run <id>`, native mana tool, and existing TUI startup/chat/session flows.

  2. Shared contracts first (47.1 -> 47.1.1/47.1.2)
     - Current evidence: imp-core/src/mana_worker.rs already re-exports tower_contracts worker types (`WorkerAssignment`, `WorkerAttempt`, `WorkerResult`, `WorkerStatus`).
     - First contract slice should formalize all cross-boundary worker shapes before further decompositions depend on ad hoc local structs.
     - Compatibility shim: keep re-exports in imp-core/mana_worker.rs until downstream call sites are migrated.
     - Rollback point: if shared-type adoption causes churn, revert to re-export layer without undoing consumer-facing behavior.

  3. Canonical worker runtime seam next
     - mana_worker.rs is already acting as the canonical single-unit worker runtime for `imp run`, native mana tool consumption, and legacy `mana run` compatibility. Make this module the explicit seam before splitting broader runtime logic.
     - Land worker-oriented module/crate extraction behind unchanged entrypoints.
     - Compatibility shim: `imp run` and legacy `mana run` continue to dispatch through the same worker runtime module.
     - Verify focus: targeted imp-core worker tests + imp-cli compile checks.

  4. Runtime loop decomposition after worker seam stabilizes (47.2)
     - Split agent.rs by turn loop, tool execution coordination, timing/events, and follow-up/continuation logic.
     - Reason for ordering: runtime extraction after worker seam avoids parallel disagreement about assignment/result handling.
     - Compatibility shim: preserve current AgentEvent/command surface while internal modules move.
     - Rollback point: extraction by internal modules first, not crate split first.

  5. Tool system decomposition after runtime loop extraction
     - Split tools/mod.rs into registry/contracts vs helper internals.
     - Keep mana tool integration stable but isolate it from unrelated helper logic.
     - Compatibility shim: re-export existing tool registry/types from current paths until consumers move.

  6. Context/session separation after runtime and tool boundaries are clearer
     - session.rs can then split along persistence/branching/replay vs title/summary/context heuristics with less churn.
     - This should trail runtime decomposition so session/context boundaries are informed by a cleaner loop.

  7. Runner/worktree/sandbox boundary integration after contracts + worker seam + runtime decomposition (47.3)
     - This is where typed runner request/result handling should replace stringly orchestration assumptions.
     - Product-layer worktree UX job 47.3.2 should trail the substrate definition enough to avoid inventing a second worktree model.
     - Compatibility shim: local execution adapter behind unchanged `imp run` / native mana behavior.
     - Rollback point: keep local adapter path intact until typed runner requests/results are proven.

  8. Capability policy and extension hardening (47.4)
     - Defer until runtime/tool/runner seams are clearer, otherwise policy work gets entangled with unstable ownership.
     - Preserve visible modes while tightening actual authority boundaries.

  9. Evidence/verification/review model (47.5)
     - Current evidence: mana_worker.rs already executes verify commands and maps close outcomes; this should evolve into a clearer evidence bundle once worker/runtime seams are stable.
     - This should not precede contract/worker cleanup because evidence artifacts belong on explicit boundaries.
     - Compatibility shim: keep current verify/close flow working while introducing structured evidence outputs.

  10. Product-layer UX and docs polish only after lower seams are stable enough
     - 47.3.2 worktree UX after runner/worktree lease model is coherent.
     - 27.4 delegation UX after native mana orchestration and worker/runtime split are clearer.
     - 28.1.5 install/onboarding/docs polish can start planning earlier, but implementation should avoid documenting unstable architecture as final.

  Shims that are acceptable during migration
  - re-exports from imp-core for shared contracts during adoption
  - temporary module facades preserving old paths while internals split
  - native mana tool / imp run dispatch adapters that call the same canonical worker runtime

  Shims to avoid
  - second independent worker runtime path for human vs orchestration mode
  - second worktree system separate from runner/worktree lease design
  - docs that imply mana/imp ownership differently from the target boundary

  Highest-risk seams needing rollback-aware slices
  - mana_worker + imp run / mana run compatibility
  - agent.rs decomposition
  - tools/mana.rs extraction from broader tools logic
  - any runner-boundary integration touching command execution/worktree behavior

  User-visible behavior that should remain stable during migration
  - `imp` remains the operator-facing Rust product
  - `imp run <id>` remains the canonical direct single-unit worker entry
  - native mana tool remains the primary orchestration UX inside imp
  - TUI remains usable; broad startup/discoverability work can trail runtime cleanup rather than leading it

In scope:
- define the migration order across contracts, imp-core decomposition, runner boundary, evidence model, and policy hardening
- define which temporary re-exports or compatibility shims are acceptable
- specify verify commands and rollback points for the highest-risk slices
- call out which user-visible behavior must remain stable during the migration

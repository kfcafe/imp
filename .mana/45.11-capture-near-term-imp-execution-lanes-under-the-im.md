  Expanded the near-term imp execution-lane plan with the implementation-detail gate: most coding details can be derived from the repo, but three clusters of consequential product/policy decisions still need explicit resolution before clean implementation sequencing.

  Decision cluster A — worker/runtime reliability
  - Auth precedence for direct/headless `imp run` when provider/model is specified but no explicit API key override is passed.
  - Whether saved OAuth/saved auth should beat saved API key and env-derived credentials.
  - Whether no-tools or shell-local startup should allow entry without model auth and defer failure until the first model-backed action.
  - How background mana completion should surface back into imp: inline assistant follow-up, notification/status only, or both.

  Decision cluster B — context efficiency and long-session safety
  - Default auto-compaction posture: disabled/manual, near-threshold automatic, or aggressive.
  - Preserved-tail guarantee for automatic compaction: what recent turns/tool-result clusters must stay verbatim.
  - Truncation bias: optimize more for lower token cost or more raw output retention.
  - Visibility policy for compaction/truncation events: always announced, quietly noted, or primarily inspect/view-visible.

  Decision cluster C — extension and script boundary
  - Confirm one-off scripting as an explicit runtime tool rather than folded into durable extension authoring.
  - Confirm Lua as current shipped extension path and TS/JS as future-facing only.
  - Default capability posture for script/guest runtimes: filesystem, network, subprocess, native-tool access.
  - Whether scripts can call native imp tools directly or only host-exposed safe APIs.
  - Durability boundary between ephemeral run-scoped scripts and durable extensions that register ordinary tools/hooks.

  Planning note: these are the main user-supplied decisions needed; file-level implementation plans, tests, module splits, and verify commands should remain repo-derived unless a later design choice makes them consequential.

Goal: make the short-horizon imp work stack legible in root mana so cross-project planning does not depend on a transient chat summary.

Current state:
- `imp/imp_rebuild_plan.md` establishes the macro rebuild order: shared contracts -> worker/runtime seam -> `agent.rs` decomposition -> tools decomposition -> context/session separation -> runner boundary -> policy hardening -> evidence-oriented outputs.
- Several immediate imp-local jobs already exist, but they are easier to misread as isolated fixes unless grouped into explicit lanes.
- The user prefers root mana as the durable planning graph when the structure affects the wider Tower rebuild.

Capture the near-term decomposition as three execution lanes:
1. Worker/runtime reliability first
   - stabilize direct headless `imp run` provider/auth resolution and stop empty-string auth fallbacks
   - make mana follow-ups/background completion delivery non-blocking so longer mana activity does not monopolize the turn
   - treat this lane as the proving slice that de-risks the worker/runtime seam before broader runner migration work
2. Context efficiency and long-session safety second
   - reduce per-turn overhead by compressing tool descriptions
   - make tool output truncation more context-aware
   - add safe automatic context compaction on top of manual `/compact`
   - treat this lane as the practical bridge between current imp session behavior and later context/session seam cleanup
3. Extension and policy boundary clarity third
   - write the guest-runtime extension substrate proposal
   - define script-tool boundaries and policy separately from durable extensions
   - treat this lane as the policy/architecture prerequisite before expanding post-Lua extension surface area

Relevant existing imp-local execution leaves to reference from root planning:
- `imp/.mana/28.1.5.1` — direct `imp run` provider auth resolution / empty-string override fix
- `imp/.mana/27.3` — non-blocking mana follow-ups and message delivery
- `imp/.mana/18` — compress tool descriptions
- `imp/.mana/21` — smarter tool output truncation
- `imp/.mana/.5` — automatic safe context compaction
- `imp/.mana/44.1` — guest-runtime extension substrate proposal
- `imp/.mana/44.2` — script tool boundaries and policy

Relationship to root rebuild plan:
- This unit does not replace root rebuild sequencing under `45.4`, `45.5`, and `45.7`.
- It records the near-term interpretation of that sequencing for active imp work: reliability -> context cost/safety -> extension boundary.
- Future root jobs should either refine one of these lanes or connect them explicitly to the formal rebuild phases.

In scope:
- durable sequencing note for cross-project planning
- mapping between macro rebuild phases and immediate imp-local leaves

Out of scope:
- implementing the leaves themselves
- changing already-defined verify gates in imp-local units
- deciding broader product/UI priorities outside these three lanes

Do not:
- duplicate the full imp-local execution specs in root mana
- treat the extension-boundary lane as permission to widen extension scope before the reliability/context lanes are stabilized

Goal: close the current production gap between the Lua capability hardening primitives and actual runtime wiring.

Current source-backed state:
- `crates/imp-lua/src/sandbox.rs` has runtime flags/allowlists (`allow_native_tool_calls`, `allowed_env`).
- `crates/imp-lua/src/bridge.rs` enforces those only if the runtime is configured, but `imp.exec`, `imp.http.*`, `imp.secret`, and `imp.secret_fields` are still exposed broadly.
- `crates/imp-lua/src/lib.rs::init_lua_extensions()` only sets native tools and loads extensions; it does not thread an explicit capability policy from `imp-core`/builder/config into the Lua runtime.
- `crates/imp-core/src/builder.rs` is still runtime-specific (`lua_tool_loader`) and does not express an extension capability model.

Do the following:
1. Define a minimal near-term capability config for the shipped Lua runtime path (no broad guest-runtime redesign here).
2. Thread it from `imp-core` / builder into Lua runtime initialization.
3. Enforce production defaults for at least:
   - `imp.tool()`
   - `imp.env()` allowlists
   - `imp.exec`
   - `imp.http`
   - secret access (`imp.secret`, `imp.secret_fields`)
4. Add focused tests proving capability-denied paths fail in real runtime wiring, not just isolated sandbox tests.
5. Keep scope tight: this is current-architecture hardening, not the full guest-runtime substrate from `44`.

- `crates/imp-lua/src/lib.rs`

Goal: record the concrete near-term implementation order for imp now that the key product/policy defaults are settled.

Current state:
- 45.11 defines the three execution lanes: worker/runtime reliability, context efficiency + long-session safety, and extension/policy boundary clarity.
- 45.11.1 now captures the settled defaults: native Rust worker execution, auth precedence and fail-fast behavior, dual completion surfacing for long mana work, and disabled-by-default auto-compaction.
- Existing imp-local units already track most of the concrete work; one new placeholder-config job under the auto-compaction epic is now added for the chosen default.

Recommended implementation order:
1. `imp/.mana/28.1.5.1` — fix direct headless `imp run` auth resolution
   Why first:
   - it fixes a concrete broken path in native Rust worker execution
   - it enforces the now-settled credential precedence and fail-fast behavior
   - it de-risks subsequent worker/runtime work because `imp run` becomes trustworthy again
2. `imp/.mana/27.3` — make mana follow-ups and completion delivery non-blocking
   Why second:
   - it completes the immediate worker/runtime reliability lane
   - it implements the chosen completion surface (notification + inline follow-up)
   - it improves live execution ergonomics without changing extension boundaries yet
3. `imp/.mana/18` — compress tool descriptions
   Why third:
   - cheapest broad context-cost win across every turn
   - small, contained change with straightforward verification
4. `imp/.mana/21` — smarter tool output truncation
   Why fourth:
   - compounds the context-efficiency gains after tool-description trimming
   - reduces prompt waste before revisiting more ambitious context-policy work
5. `imp/.mana/.5.1` — add disabled-by-default auto-compaction config scaffolding
   Why fifth:
   - matches the settled policy of no automatic compaction yet
   - creates the future config seam without taking on runtime behavior risk
6. `imp/.mana/44.1` — author guest-runtime extension substrate proposal
   Why sixth:
   - extension architecture should be clarified only after the native Rust worker/runtime path is stabilized
   - proposal must explicitly preserve worker execution as Rust-native and Lua as the current shipped extension path
7. `imp/.mana/44.2` — define script tool boundaries and policy
   Why seventh:
   - script tool policy should follow the substrate framing
   - worker execution has now been explicitly separated from script/Lua surfaces

Execution notes:
- Treat items 1 and 2 as the mandatory first lane.
- Items 3, 4, and 5 form the second lane and can be adjusted slightly for tactical reasons, but the preferred order is cost reduction before placeholder compaction settings.
- Item 7 should follow item 6 because the script-tool policy depends on the substrate/boundary framing.

In scope:
- near-term sequencing only
- mapping settled defaults to concrete imp-local units

Out of scope:
- replacing formal root rebuild phases
- implementation itself
- broader CLI/TUI product-priority reordering outside these lanes

  2026-04-27T03:58:49.934887+00:00
  Audit update from local `/Users/asher/.pi/agent/extensions`:
  - Pi extension source exists under user config, not project-local: `~/.pi/agent/extensions`.
  - Shape is mixed: 11 top-level `.ts` extension files plus 25 extension directories.
  - Directory packages commonly use `package.json` with `pi.extensions: ["./index.ts"]`; examples: `web`, `database`, `test-runner`, `8b-tools`, `diff`, `file-watcher`, `lazy-tools`, `multi-edit`, `vibecheck`, `youtube`. One notable exception found: `mana/package.json` lacks a `pi` field despite having `index.ts`.
  - Single-file extensions include `ask.ts`, `ast-grep.ts`, `bn-issue.ts`, `cron.ts`, `files.ts`, `pdf-reader.ts`, `venice-image.ts`, etc.
  - Runtime API pattern: default exported function accepts `ExtensionAPI` from `@mariozechner/pi-coding-agent`; extensions call `pi.registerTool({...})`, `pi.registerCommand(name, {...})`, and `pi.on(event, handler)`.
  - Tool schemas commonly use `@sinclair/typebox` `Type.*` values as `parameters`.
  - Dependency story is package-local npm dependencies in some extension dirs (`web` uses jsdom/turndown/playwright/readability; `database` uses pg/better-sqlite3; watcher/diff use chokidar/diff), while single-file extensions rely on Pi-provided packages and node/bun builtins.
  - Events/hooks in current extensions include `session_start`, `session_shutdown`, `session_switch`, `message_end`, `agent_end`, `tool_result`, `tool_call`, `before_agent_start`, and `context`; some regex false positives from child processes (`data`, `close`, `error`) must be ignored in any static scanner.
  - Existing imp import CLI is in `crates/imp-cli/src/lib.rs::run_import`; core detection/import is in `crates/imp-core/src/import.rs`. It currently detects Pi skills and AGENTS.md only, then copies skills into `~/.config/imp/skills`.
  Imp implication: next implementation should extend the existing import path with Pi extension detection/copying instead of inventing `imp extension import`. First pass can copy TS extension packages into an imp extensions staging directory and report them as imported-but-not-executable until TS guest runtime exists.

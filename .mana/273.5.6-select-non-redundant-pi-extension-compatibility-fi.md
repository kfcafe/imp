Goal: choose TypeScript extension compatibility fixtures deliberately instead of blindly importing all local Pi extensions.

User direction:
- Current local Pi extensions may not be the best tools to import because imp already has strong native tools.
- `color-palette` is a possible local smoke target because it is self-contained.
- Look for other cool Pi-compatible tools online that exercise the compatibility layer and are not redundant with imp-native tools.

Steps:
1. Audit local `~/.pi/agent/extensions` for low-dependency, self-contained candidates and classify whether each is redundant with imp-native tools.
2. Search online for Pi extensions/examples/packages compatible with `@mariozechner/pi-coding-agent` or the Pi extension API.
3. Select a small fixture set:
   - one local low-dependency smoke fixture, likely `color-palette` if still suitable
   - one external/simple extension fixture that proves import/execution compatibility outside Asher's local toolset
   - optionally one package-with-dependencies fixture if runtime dependency loading is in scope
4. Record why each fixture was chosen and what API surface it covers: `registerTool`, TypeBox schemas, command registration, hooks, package deps, UI, env/network/process.
5. Feed selected fixtures back into implementation/test tasks for the sprint.

Out of scope:
- Importing every local Pi extension.
- Prioritizing tools already covered better by imp native tools.

Acceptance:
- The sprint has a documented compatibility fixture set with rationale and coverage, including at least one non-redundant or external/online Pi extension candidate.

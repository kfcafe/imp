Goal: separate generic imp TypeScript extension runtime responsibilities from Pi compatibility responsibilities so the architecture does not become a Pi-only runtime.

Current reason:
- The current `typescript_extensions.rs` spike proves the concept but mixes generic runtime, Pi package discovery, Pi `ExtensionAPI` shim, Bun bridge script, schema normalization, and fixture tests.
- Refactor before adding more compatibility so future non-Pi TypeScript extensions can fit without pretending to be Pi packages.

Target module seams:
1. `typescript_extensions/mod.rs`
   - public entrypoint
   - extension discovery under `.imp/extensions`
   - host registry loading
   - generic types exported to builder/tests
2. `typescript_extensions/bun_runner.rs`
   - Bun availability check
   - bridge file creation/execution
   - inspect/execute calls
   - timeout/error normalization later
3. `typescript_extensions/pi_compat.rs`
   - Pi package shape and `package.json.pi.extensions`
   - Pi `ExtensionAPI` shim/prelude
   - static import rewrites for Pi package imports
   - lifecycle/API stubs (`registerCommand`, `pi.on`, `ctx.ui`, `sessionManager`)
4. `typescript_extensions/schema.rs`
   - TypeBox/JSON schema normalization
   - provider-compatible restrictions and actionable errors
5. `typescript_extensions/tests.rs` or fixture module
   - synthetic `registerTool` fixture
   - local `color-palette` fixture
   - official/reduced `dynamic-tools` fixture

Behavior-preservation requirement:
- Refactor only; keep current 8 TS extension tests green with Bun.
- Do not expand API compatibility during the extraction unless required to keep existing tests passing.

Acceptance:
- The code has a named seam between generic TypeScript runtime and Pi compatibility.
- Tests distinguish generic TS extension behavior from Pi compatibility behavior.
- `PATH="$HOME/.bun/bin:$PATH" cargo test -p imp-core typescript_extensions -- --nocapture` remains green.

## Goal
Turn the completed architecture note in `docs/design/code-intelligence-strategy.md` into an explicit follow-on plan with clear slices, ownership, and durable deliverables.

## Current state
- `29.2` defines the architectural boundary: `imp` owns live code-intelligence surfaces; `mana` owns only promoted durable consequences.
- The note recommends phased rollout: read-oriented semantic inspection first, guarded write-oriented semantic actions later, and selective durable promotion into `mana`.
- No explicit root mana decomposition exists yet for those follow-on planning slices.

## Scope
- Root-scope planning and decomposition only.
- Clarify implementation slices that may later land in `imp` and, where needed, shared/durable evidence contracts touching `mana`.
- Keep `mana` out of live LSP/runtime orchestration.

## Deliverable
- Child units that separately plan:
  1. read-oriented `imp` code-intelligence surfaces,
  2. guarded write-oriented semantic actions and policy,
  3. durable evidence/promotion rules for diagnostics and verify results,
  4. the internal `imp` subsystem/crate boundary.

## Important constraints
- Do not treat every tool output as durable memory.
- Do not collapse AST tools, LSP, and script execution into one abstraction.
- Do not assign live language-server orchestration to `mana`.

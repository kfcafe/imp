  Principle:
  - do not migrate TUI slash commands as a flat bag of literals; reorganize them into coherent CLI capability clusters and expose them across shell commands, direct CLI subcommands, and optional viewer/TUI surfaces.

  Proposed capability clusters and UX shape:
  1. session commands
     - current sources: new, resume, fork, name, export, tree/session alias
     - shell commands: `:new`, `:resume`, `:fork`, `:name <name>`, `:export [path]`, `:tree`
     - direct CLI surfaces to evaluate: `imp sessions`, `imp export`, `imp resume`
     - browsing-heavy variants may hand off to `imp view`
  2. runtime controls
     - current sources: model selector, thinking cycle, compact, reload, quit
     - shell commands: `:model [name]`, `:thinking [level]`, `:compact`, `:reload`, `:quit`
     - no fullscreen dependency; fast inline control preferred
  3. setup / config / identity
     - current sources: settings, personality, login, secrets, setup/welcome
     - shell commands: `:settings`, `:personality`, `:login [provider]`, `:secrets [provider]`, `:setup`
     - support quick subcommands for common edits and guided CLI flows for heavier edits
     - optional richer TUI/editor flow may remain for browsing/editing
  4. memory + checkpoints
     - shell commands: `:memory ...`, `:checkpoints`, `:restore-checkpoint <id>`
     - also strong candidates for direct CLI subcommands (`imp memory`, `imp checkpoints`)
  5. utility / help / copy / file inclusion
     - shell commands: `:help`, `:copy [target]`, `:files <query>` or similar
     - preserve `@file` include syntax where useful, but replace TUI-only file picker with CLI search/select patterns
  6. extension commands
     - Lua commands must remain discoverable and callable from the shell namespace
     - add a discoverability surface such as `:commands`

  Syntax recommendation:
  - prefer colon commands in the CLI shell (`:model`, `:settings`, etc.)
  - accept legacy slash-command compatibility during transition if needed
  - reserve `imp view` / `imp tui` for browsing-heavy or richer UI tasks rather than making them default dependencies of the shell

  User-facing rule:
  - simple controls inline, complex edits in guided CLI flows, browsing-heavy tasks in viewer/TUI

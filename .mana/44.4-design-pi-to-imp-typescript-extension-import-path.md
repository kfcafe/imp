Goal: define the next concrete compatibility step for making imp able to import user TypeScript extensions from Pi without prematurely committing imp to a TS-first runtime rewrite.

Context from current architecture docs:
- `docs/proposals/guest-runtime-extension-substrate.md` says Rust remains the host authority, Lua is the current shipped guest runtime, and TypeScript/JS should be added only after a host-neutral extension substrate, manifest, and capability model exist.
- `docs/proposals/script-tool-boundaries-and-policy.md` keeps one-off script execution separate from durable extensions.
- User specifically wants users to easily import their extensions from Pi into imp.

Scope:
1. Inspect Pi extension packaging/import shape and identify the smallest common manifest/metadata contract imp can understand.
2. Define a compatibility manifest for imp extension packages that can represent Pi TypeScript extensions and existing Lua extensions.
3. Decide whether first support should be import-only metadata/packaging validation or executable TS runtime support.
4. Produce a short implementation breakdown for: `imp extension import <path-or-pi-ref>`, package copy/link behavior, manifest validation, capability prompt, and eventual TS guest-runtime adapter.

Out of scope:
- Implementing a JS/TS engine before the host manifest and capability contract are stable.
- Replacing Lua.
- Routing worker execution through TypeScript.

Likely files:
- `docs/proposals/guest-runtime-implementation-plan.md`
- `docs/proposals/guest-runtime-extension-substrate.md`
- `crates/imp-core/src/builder.rs`
- `crates/imp-core/src/tools/extend.rs`
- `crates/imp-lua/src/*`

Acceptance:
- There is a concrete documented import-first path for Pi TypeScript extensions into imp, including manifest fields, import UX, capability policy, and the dependency order before executable TS support.

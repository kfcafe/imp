  Current code/doc facts to preserve:
  1. `run_headless_mode()` already switches between `HeadlessOutputMode::Human` and `HeadlessOutputMode::Json` based on TTY / explicit protocol-ish modes.
  2. `imp-cli` currently has duplicated machine encoders: `print_json_event()` + `stream_event_to_json()` for headless/run and separate `rpc_agent_event_to_json()` + `rpc_stream_event_to_json()` for RPC.
  3. Those machine encoders already diverge in field shape (`tool` vs `tool_name`, flattened RPC tool-end payload vs nested headless result, different error/event envelopes), so this slice must stabilize one contract before later affordances build on it.
  4. Human shell transcript expectations are already documented: compact transcript, terse tool notices, inline asks/confirms, summary line, and deeper inspection moved to `imp view`.
  5. Shared UI request semantics already exist in `imp_core::ui::UserInterface`; TUI is an adapter over that seam via `tui_interface.rs`, not the owner of fullscreen-only semantics.
  6. `imp view` is a browse-heavy human surface (`sessions/tree/logs/checkpoints`), not the baseline machine-stream surface.
  7. The worker-runtime proposal already points toward `imp run --protocol worker-json` as a machine-facing profile and explicitly warns against duplicate protocol stacks.

  Resolved planning target:
  A. Default surface orientation
  - Human-by-default: `imp chat`, `imp view`, `imp tui`
  - Machine-by-default: RPC / JSON protocol mode
  - Conditional/default-switching: `imp run` / headless worker path should default to human transcript when stdout is a TTY and no explicit machine protocol is requested, but default to machine JSON/JSONL when piped or when an explicit protocol/json mode is selected.

  B. Shared substrate below all surfaces
  - The canonical shared runtime substrate remains `AgentEvent`, `StreamEvent`, and the shared UI-request categories.
  - Human rendering and machine serialization should both sit on top of that shared substrate rather than inventing surface-local event families.
  - Later affordances (checkpoints, planning, approval, detached execution) should depend on this shared event/result contract instead of adding new output styles.

  C. Ownership boundary
  - `imp-core` should own the canonical event/result/request types and any surface-neutral structured summaries needed across surfaces.
  - Thin adapters should own presentation only:
    - CLI shell transcript formatting
    - `imp view` browse formatting
    - TUI fullscreen presentation and local interaction state
    - RPC transport wrappers / stdin-stdout control loop
  - Do not let TUI or RPC define new canonical runtime semantics.

  D. Machine-facing contract direction
  - `imp run --protocol worker-json` should be a constrained profile over the same canonical machine event stream used by headless JSON, not a parallel schema.
  - RPC should extend that same event family with transport/control envelopes like `ui_request`, `ui_response`, and `protocol_error`, rather than diverging in field names or per-event shapes.
  - Exact envelope versioning and field names can be refined later, but this unit resolves the boundary: one canonical machine event family, multiple surface adapters/profiles.

  E. Human-facing contract direction
  - Human output should stay transcript-oriented and compact by default: streamed assistant text, terse tool notices, actionable warning/error expansion, turn summary, status-row-driven shell context.
  - Human browse-heavy inspection should route to `imp view` rather than dumping large payloads inline.
  - TUI remains a richer human adapter over the same runtime events and shared UI requests, not a separate output contract.

  F. Concrete next implications for follow-on units
  - `50.16.2` and later CLI-native affordance units should assume this split instead of restating it.
  - `50.14` should define the shared UI request + runtime-event seam that both human rendering and machine serialization consume.
  - `248.18.3` should refine streamed-error structure as a sub-problem inside this single machine-facing contract.
  - Implementation follow-up should merge `print_json_event()` / `stream_event_to_json()` and `rpc_agent_event_to_json()` / `rpc_stream_event_to_json()` behind one canonical serializer layer, then let headless/worker/RPC select profiles rather than duplicate encoders.

  Open detail intentionally deferred, not unresolved at the boundary level:
  - exact machine envelope version field and compatibility policy
  - whether `ToolOutputDelta` becomes part of the canonical machine stream or remains human/TUI-local by schema design
  - whether bare `imp chat` should ever gain explicit machine export flags, which is lower priority than stabilizing `imp run` + RPC first

## Task
Create the durable planning target for the first CLI-first affordance slice: stable human vs machine output modes across `imp chat`, `imp run`, RPC, `imp view`, and `imp tui`.

## Why this slice is first
Per `docs/rebuild/imp-cli-affordance-sequence.md`, output-mode stability is the enabling layer for later checkpoints, planning, approval, and detached execution. It must follow the shared-runtime startup map rather than adding more surface-specific render paths.

## Grounding
- `docs/rebuild/imp-cli-affordance-sequence.md`
- `docs/rebuild/imp-shared-runtime-startup-map.md`
- `docs/rebuild/imp-cli-shell-transcript-ux.md`
- `imp/docs/proposals/imp-run-worker-contract-and-mana-run-handoff.md`

## Questions this unit should pin down
1. Which surfaces are human-oriented by default versus machine-oriented by default?
2. What shared event/result model should sit underneath human rendering and machine serialization?
3. Which parts of rendering belong in shared `imp-core` versus thin surface adapters?
4. How should `imp run --protocol ...` align with RPC/headless JSON output without keeping duplicate encoders?

## Acceptance
Leave root mana with an explicit durable planning target for the output-mode contract so later affordance units can depend on it rather than restating the same boundary.

  Repo-grounded contract recommendation after inspecting `mana-core/src/ops/close.rs`, `mana-review/src/types.rs`, `tower-contracts/src/lib.rs`, `imp-core/src/mana_worker.rs`, and worker-status consumers:

  1. Minimal v1 verifier/evidence additions should stay explicitly execution-lineage oriented.

  Recommended new shared types next to the worker quartet:
  - `VerifierStatus` enum: `Passed | Failed | Skipped`
  - `ArtifactKind` enum: start narrow with `VerifyOutput`, `DiffScopeSummary`, `Patch`, `ReviewRecord`, `Log`, `Other`
  - `ArtifactRef`
  - `VerifierResult`
  - `EvidenceBundleRef` (reference shape first, not a fully embedded bundle struct)

  2. Prefer reference-first evidence over a full embedded `EvidenceBundle` struct in the first implementation slice.
  Reason:
  - current repo already computes rich durable evidence in mana (`CloseEvidence`, `VerifyFailureResult`, review/risk records)
  - first implementation value is making lineage legible, not prematurely centralizing all evidence payloads in `tower-contracts`
  - a reference-first shape avoids duplicating storage semantics before the durable bundle layout is settled

  3. Suggested minimal field sets.

  `ArtifactRef`
  - `artifact_id: String`
  - `kind: ArtifactKind`
  - `locator: String`  // path/URI/reference token; name chosen generically to avoid overcommitting to storage backend
  - `run_id: Option<String>`
  - `unit_id: Option<String>`
  - `stage: Option<String>`

  `VerifierResult`
  - `verifier_name: String`
  - `status: VerifierStatus`
  - `command: Option<String>`
  - `exit_code: Option<i32>`
  - `summary: Option<String>`
  - `artifact_refs: Vec<ArtifactRef>`
  - `started_at: Option<String>`
  - `finished_at: Option<String>`
  - `run_id: Option<String>`
  - `unit_id: Option<String>`

  `EvidenceBundleRef`
  - `bundle_id: String`
  - `unit_id: String`
  - `run_id: Option<String>`
  - `artifact_refs: Vec<ArtifactRef>`
  - `summary: Option<String>`

  4. Ownership split these types should preserve.
  - `imp` produces immediate runtime outcome + verifier results it directly executes.
  - `mana` persists durable close/verify/review artifacts and can mint the canonical `EvidenceBundleRef` or attach refs to approval/close records.
  - `tower-contracts` owns the cold-reference vocabulary so imp/mana/runner/reviewer can all point at the same lineage without sharing storage internals.

  5. Mapping guidance from current repo types.
  - `mana_worker::run_verify_command()` and `map_verify_failed_close_outcome()` show the current raw materials for `VerifierResult`.
  - `mana-core::VerifyFailureResult` maps naturally to a failed `VerifierResult` plus attached verify-output artifact ref.
  - `mana-core::CloseEvidence` should initially stay mana-local and be referenced by `ArtifactRef { kind: DiffScopeSummary, ... }` rather than moved wholesale into `tower-contracts` in the first slice.
  - `mana-review::Review` should also stay local and be referenced by `ArtifactRef { kind: ReviewRecord, ... }`.

  6. Compatibility adapter plan for the first implementation slice.
  - Add the new shared types in `tower-contracts` alongside the existing worker module.
  - Keep `imp_core::mana_worker` as the first adapter seam that can construct `VerifierResult` values without changing outer behavior.
  - Avoid renaming `WorkerResult` to `WorkerOutcome` in the same slice.
  - Convert mixed consumers gradually: code that only needs worker/evidence contracts should import from `tower_contracts` directly; execution helpers remain in `mana_worker`.

  7. Focused verify guidance for the first implementation slice.
  - `cd /Users/asher/tower && cargo check -p imp-core -p imp-cli`
  - targeted tests around `mana_worker` close/verify mapping and any new `tower-contracts` serialization/shape tests

  8. Anti-creep guardrail.
  Do not move `CloseEvidence`, `CloseResult`, `VerifyFailureResult`, or `Review` wholesale into `tower-contracts` in this slice. The first slice should add shared lineage vocabulary, not flatten all durable evidence ownership into the contracts crate.

Goal: define and enforce the sprint-1 TypeScript/Pi extension support boundary so imported extensions do not imply full Pi parity.

Durable status model:
1. `executable`
   - Extension loads and can register/execute tools now.
2. `loaded_with_stubs`
   - Extension loads and works, but touched APIs currently stubbed/recorded by imp compatibility layer, e.g. `registerCommand`, `ui.notify`, `ui.setStatus`, `sessionManager.getBranch/getEntries`.
3. `needs_dependencies`
   - Extension appears otherwise loadable, but installable npm/Bun dependencies are missing. This is not unsupported. The user should be notified and asked before installation.
4. `unsupported`
   - Extension requires APIs/runtime capabilities imp does not currently support, e.g. `ctx.ui.custom`, rich viewer/rendering mapping, built-in tool overrides, unrestricted filesystem/network/process/env/secrets/native tool access.

Dependency UX policy:
- Assume local Pi extension dependencies are already installed/resolvable when importing.
- If missing dependency errors occur during validation/load, classify as `needs_dependencies`.
- Notify the user with the missing dependency/error and ask whether to install with Bun.
- Never silently install dependencies during import.
- Reserve `unsupported` for non-installable compatibility gaps.

Implementation requirements:
1. Update `TypeScriptExtensionLoadState` to include `NeedsDependencies` / equivalent.
2. Teach Bun bridge/status inspection to identify missing dependency errors separately from unsupported runtime/API errors.
3. Update `imp import` runtime status output to print `needs dependencies` and an install prompt path.
4. Add a missing-dependency fixture/test that asserts the state is `needs_dependencies`, not `unsupported`.
5. Keep existing executable/stubbed fixture tests green.

Acceptance:
- Users can distinguish executable, stubbed, missing-dependency, and unsupported extension states.
- Missing deps are handled as installable dependency needs with explicit user confirmation, not unsupported failures.

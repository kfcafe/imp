Goal: specify and implement the first user-facing import workflow for bringing Pi extensions into imp by extending the existing `imp import --from pi` path, not by adding a new top-level command.

User clarification:
- Existing Pi extensions are locally available under `~/.pi/agent/extensions`.
- imp already has an import CLI command that imports from Pi; extension import should be added to that existing command.

Current imp import seam:
- CLI: `imp/crates/imp-cli/src/lib.rs::run_import`
- Core detection/import: `imp/crates/imp-core/src/import.rs`
- Current behavior imports Pi skills and AGENTS.md only.

First-pass desired behavior:
1. `imp import --from pi --dry-run` detects Pi extensions alongside skills/instruction files.
2. Detection covers:
   - `~/.pi/agent/extensions/*.ts` single-file extensions
   - `~/.pi/agent/extensions/*/package.json` where `pi.extensions` lists entrypoints
   - fallback directories with `index.ts` even if package metadata is incomplete, e.g. current `mana` package shape
3. Import copies or stages extensions into an imp-owned user config location, recommended initial target: `~/.config/imp/extensions/pi/<name>/...`.
4. Existing skip semantics are preserved for already-imported names.
5. User output reports imported/skipped extensions separately from skills and AGENTS.md.
6. Since imp does not yet execute TypeScript guest extensions, successful import must be honest: imported/staged but not executable until TS runtime support lands.
7. Tests cover detection and copy behavior for single-file, package-json, fallback index.ts, and already-existing extension cases.

Out of scope for this unit:
- Implementing executable TypeScript guest runtime support.
- Replacing Lua extension loading.
- Adding a brand-new `imp extension import` command.

Acceptance:
- The import UX is specified enough that a coding agent can implement command parsing, package placement, validation, and unsupported-runtime messaging without new product decisions.

Goal: wire imp-core’s existing FileHistory safety primitive into a real checkpoint foundation without changing overall architecture.
Current state: crates/imp-core/src/tools/mod.rs defines FileHistory with snapshot_before_edit(), original(), rollback(), and tracked_files(), but Agent/ToolContext only carry FileCache + FileTracker. write/edit/multi_edit do not snapshot via a shared session-level history object, so checkpoint state is not accumulated during real runs.

Steps:
1. In crates/imp-core/src/tools/mod.rs add a session-scoped checkpoint/history holder around FileHistory that can accumulate named checkpoint metadata (checkpoint id, optional label/summary, tracked files, created_at). Keep it small and local-first.
2. Extend ToolContext and Agent to carry shared checkpoint/file-history state alongside file_cache and file_tracker.
3. In crates/imp-core/src/agent.rs pass the shared checkpoint/file-history state into ToolContext.
4. In crates/imp-core/src/tools/write.rs, edit.rs, and multi_edit.rs snapshot existing files before destructive writes using the shared history object. Reuse the current first-edit-wins semantics; do not redesign the edit architecture.
5. Add a minimal API on the shared checkpoint state to expose the currently tracked files/checkpoints for later session persistence and restore work.
6. Add focused tests proving real tool execution populates shared checkpoint/file-history state and preserves rollback behavior.

Files:
- crates/imp-core/src/tools/mod.rs (modify — add shared checkpoint/file-history state and tests)
- crates/imp-core/src/agent.rs (modify — add shared state to Agent and ToolContext wiring)

- crates/imp-core/src/tools/edit.rs (modify — snapshot before edit)

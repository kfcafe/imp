  Current state after `.3.2`:
  - Route/model selection is now shared between `imp-cli` and `ImpSession::create()`.
  - Remaining auth issues are isolated to credential handling rather than provider-route selection.
  - Observed problem points:
    1. `AuthStore::set_runtime_key()` accepts empty/whitespace strings, so a blank override can survive as state.
    2. `ImpSession::create()` still had `unwrap_or_default()` in the no-tools/headless path, which turns auth failure into `api_key=""` and caused the earlier `Incorrect API key provided: ''` behavior.

  Execution plan for this unit:
  1. Normalize empty/whitespace runtime API-key overrides to absence in `crates/imp-llm/src/auth.rs` by making `set_runtime_key()` trim values and remove the runtime override when the provided string is empty.
  2. Make auth resolution model/route-consistent in `crates/imp-core/src/imp_session.rs`:
     - for `openai-codex`, continue using `resolve_chatgpt_oauth()`
     - for `anthropic`, continue using `resolve_with_refresh()`
     - for other resolved routes (including plain `openai`), use `resolve(...)` so saved auth is preferred before stored/env API keys
  3. Remove the no-tools/headless `unwrap_or_default()` fallback so missing auth fails clearly instead of becoming an empty bearer token.
  4. Align the thin CLI auth helper in `crates/imp-cli/src/lib.rs` with the same route-aware behavior so operator/headless paths do not fork again.
  5. Add/adjust focused tests for:
     - empty runtime key normalization
     - `resolve("openai")` preferring saved OAuth while `resolve_api_key_only("openai")` still ignores OAuth
     - session/headless auth failure behavior now surfacing an explicit error instead of falling through to `""`
  6. Verify with the narrowest relevant commands first (`imp-llm` auth tests, `imp-core` imp_session tests, and focused `imp-cli` auth/model tests) before broader checks.

  Boundary for this unit:
  - This is auth precedence and empty-string cleanup only.
  - Do not reopen provider-route selection logic from `.3.2` unless a test exposes a real regression on the new shared resolver path.

Goal: make headless worker auth clean and predictable by normalizing empty API-key overrides to absent values, resolving credentials through one precedence order, and failing clearly instead of sending empty tokens.

Current state:
- Empty or missing auth in the no-tools/headless path can collapse into an empty string token.
- The worker/session path special-cases OpenAI auth in ways that bypass saved OAuth.
- The user wants auth resolution to check saved auth first, then API-key sources.

Steps:
1. Normalize empty/whitespace API-key overrides to `None` at the CLI/session boundary.
2. Refactor runtime auth resolution so it uses one precedence order for the chosen route: explicit non-empty override -> saved auth -> stored API key -> env -> clear error.
3. Remove any auth-failure-to-empty-string fallback from headless/no-tools worker startup.
4. Ensure OpenAI/ChatGPT saved auth is considered where the chosen route can use it.
5. Add focused tests for empty-string normalization, saved-auth preference, and explicit failure behavior.

Files:
- `crates/imp-cli/src/lib.rs`
- `crates/imp-core/src/imp_session.rs`
- `crates/imp-llm/src/auth.rs`

In scope:
- auth precedence and normalization for headless workers
- explicit error behavior instead of empty token fallthrough

Out of scope:
- broader credential-store redesign

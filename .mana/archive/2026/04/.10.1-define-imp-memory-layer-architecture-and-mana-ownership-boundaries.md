## Goal
Turn the current vague "memory consolidation" discussion into one explicit architecture document. Right now imp has several overlapping memory mechanisms — session history with compaction, frozen per-session personal memory snapshots, mana facts/task context, and the emerging idea of a mana-backed wiki layer — but there is no single canonical statement of what each layer owns.

## Scope
- In scope: define the memory layers, their responsibilities, their persistence model, and the read/write boundaries between imp runtime and mana.
- In scope: explain what belongs in session history, what belongs in `memory.md` / `user.md`, what belongs in mana units/facts, and what belongs in a future synthesized wiki layer.
- Out of scope: implementing the wiki, adding new storage backends, or changing provider/session semantics in code.

## Deliverable
- A proposal doc at `docs/proposals/imp-memory-architecture-and-mana-boundary.md`.
- If useful, add brief cross-links in `README.md` or `ARCHITECTURE.md`, but keep the implementation strictly docs-first.

## Patch Guidance
- Docs-only patch.
- Keep it concrete and operational, not philosophical.
- Reuse inspected runtime behavior from the cited files instead of inventing a new subsystem.

## Focus
- `LEARNING_LOOP_SPEC.md` (personal memory model and frozen snapshot pattern)
- `crates/imp-core/src/session.rs` (session persistence + compaction boundary semantics)
- `crates/imp-core/src/imp_session.rs` (model-visible active history and prefill injection)
- `crates/imp-core/src/system_prompt.rs` and `builder.rs` (prompt-layer assembly)
- `crates/imp-core/src/config.rs` (mana-first role doctrine)
- `README.md` and `ARCHITECTURE.md` (public architectural framing)

## Required Questions to Answer
1. Which memory is session-local vs cross-session vs project-durable?
2. Which layer is authoritative for project knowledge?
3. What should never be pushed into imp session state if mana can hold it durably?
4. How should agents reason about "facts" vs "synthesis" vs "preferences"?
5. What are the anti-goals (for example: do not turn imp memory.md into a project wiki)?

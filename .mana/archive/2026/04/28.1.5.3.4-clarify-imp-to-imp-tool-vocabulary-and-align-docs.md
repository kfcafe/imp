  2026-04-15T07:36:18.834352+00:00
  Inspection findings before renaming the native imp tool contract:
  - `crates/imp-core/src/tools/imp.rs` currently exposes user-facing `delegate` everywhere: tool label (`Delegate Work`), tool description, schema action enum (`["delegate"]`), runtime validation/error text, helper function names, output detail fields (`action: delegate`, `delegation_mode`), and tests.
  - README/operator docs also still use `delegate` / `delegating` language in several places, especially around orchestrator mode and mana child jobs.
  - The current root/proposal direction says the CLI namespace should likely converge on `imp mana ...` for operator-facing execution, so the native imp tool should avoid `run` to prevent another collision.
  - Strong candidate verb after inspection: `spawn` (clear child-worker meaning, distinct from CLI `imp mana <id>`, works for both `unit` and `ad_hoc` modes). Likely contract shape: preferred `action='spawn'` with `spawn_mode` or `mode`; keep `action='delegate'` as backward-compatible alias during migration.
  - Alternative candidates still possible but weaker: `handoff` (less literal for transient helper execution), `worker` (noun-like and awkward as an action), `assist` (too broad), `dispatch` (too orchestration-heavy and close to mana scheduling ownership).
  Need one naming decision before patching the tool contract end-to-end.

Goal: write the crisp architectural specification for the strengthened `imp run` worker/runtime contract and its relationship to `mana run`.

Current state:
- We have a clear product direction: keep `mana run` as the multi-unit parallel orchestrator, but strengthen `imp run` so orchestration rests on a first-class worker runtime rather than the current thin headless path.
- Existing code evidence: `imp/crates/imp-cli/src/main.rs` manually loads units and shells out to `mana close`, while `imp/crates/imp-core/src/tools/mana.rs` already has richer native mana run-state/status UX.
- Archived unit `29.7` already decided not to add a new native imp orchestration tool action. This spec must preserve that boundary and explicitly distinguish “strengthen `imp run` CLI/runtime contract” from “add a new imp orchestration substrate.”

Steps:
1. Write a concise spec that states the desired architecture and ownership split: mana orchestrates many units; imp executes one unit well.
2. Define the target `imp run` worker contract, including flags/modes such as explicit mana dir, machine-readable protocol, and defer-verify semantics.
3. Define the outcome protocol and handoff semantics so close/verify ownership stays with mana in orchestrated runs.
4. Define what should change in the human-facing `imp run <id>` UX versus machine-facing worker mode.
5. Capture non-goals and migration phases so later implementation jobs do not invent orchestration inside imp.

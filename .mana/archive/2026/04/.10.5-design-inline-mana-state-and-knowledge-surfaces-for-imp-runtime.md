## Goal
Define how imp should surface mana state and future synthesized knowledge inline between turns so the user can actually see the durable work graph evolving. The current epic explicitly calls for UI/runtime affordances here, and there is already related work in units `27.2` (compact mana status/progress surface) and `27.3` (non-blocking mana follow-ups and message delivery), but no cohesive design tying those efforts to the new memory architecture.

## Current State
- `27.2` proposes a compact mana status widget in the TUI but is blocked on `27.1` and `30`.
- `27.3` proposes non-blocking mana follow-ups and background completion but is still in progress (1 abandoned attempt).
- The TUI sidebar already has mana-specific formatting in `views/sidebar.rs` and `views/tools.rs` (format_mana_output, format_mana_args).
- No current surface shows synthesized project knowledge or wiki-derived summaries.

## Scope
- In scope: specify minimal UI/runtime surfaces for mana status, newly created child jobs, relevant facts/wiki summaries, and background completion notifications.
- In scope: map this design onto existing work in `27.2` and `27.3` so we reuse current mana progress/follow-up plumbing.
- Out of scope: implementing the UI itself in this unit.

## Deliverable
- A proposal doc at `docs/proposals/inline-mana-state-and-knowledge-surfaces.md`.
- The doc should identify the smallest useful UI surfaces for a first implementation and which existing units they extend.

## Patch Guidance
- Docs-only patch.
- Avoid inventing a huge dashboard; prefer small, reviewable surfaces that fit imp current TUI.
- Distinguish raw work-state signals from synthesized knowledge summaries.

## Focus
- `crates/imp-tui/src/app.rs`
- `crates/imp-tui/src/views/sidebar.rs`
- `crates/imp-tui/src/views/tools.rs`
- `crates/imp-core/src/tools/mana.rs`
- Existing related units: `27.2` and `27.3`

## Required Questions to Answer
1. What should be visible immediately after imp creates or updates mana records during a conversation?
2. What should surface only on completion/notification paths?
3. How should between-turn visibility work without flooding the transcript?
4. How should future wiki/synthesis summaries be distinguished from raw mana status?
5. Which parts should extend `27.2` / `27.3` versus becoming new follow-on work?

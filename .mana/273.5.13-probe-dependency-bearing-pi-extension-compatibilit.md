Goal: test dependency-bearing Pi extension compatibility and define dependency policy, without assuming a specific extension such as `youtube`.

User direction:
- Since `imp import` already imports skills from Pi, it should naturally import Pi extensions alongside them.
- For dependencies: assume deps are already installed when importing local Pi extensions.
- If dependencies are missing, do not classify the extension as unsupported. Notify the user about missing deps and ask whether to install them.
- Unsupported should be reserved for APIs/features imp cannot support, not missing installable dependencies.

Candidate selection:
1. Audit local `~/.pi/agent/extensions/*/package.json` dependencies and runtime assumptions.
2. Pick a modest dependency-bearing extension based on API/dependency shape, not because of any specific name.
3. Avoid large/high-risk first candidates unless specifically chosen for dependency UX testing.
4. Prefer a fixture that tests dependency detection/install prompting without adding unrelated API complexity.

Dependency policy to validate:
- Import copies/stages the extension by default as part of normal `imp import` behavior.
- Runtime status assumes existing local deps are present and uses Bun to resolve them.
- Missing dependencies produce a `needs dependencies` / `dependencies missing` state with a clear package/error message.
- CLI/user flow asks before installing missing deps.
- No silent dependency install during import.

Out of scope:
- Auto-installing dependencies without explicit user confirmation.
- Full compatibility for all dependency-heavy extensions.

Acceptance:
- A dependency-bearing Pi extension has a status/fixture test or documented smoke result.
- The status model distinguishes missing deps from truly unsupported APIs.
- The sprint has a clear dependency policy: assume installed, ask before installing missing deps.

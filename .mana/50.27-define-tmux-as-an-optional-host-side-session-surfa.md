Goal: capture a concrete architecture/decomposition for whether and how imp should integrate with tmux without letting tmux become the core runtime boundary.

Current state:
- Existing root work already covers detached/background local execution contracts (`50.25`) and the first local detached/background execution model (`50.26`).
- A planning conversation concluded that tmux could be useful for persistent human-observable worker sessions, operator takeover/debugging, and lightweight stopgap process hosting, but should not become imp's durable memory, orchestration substrate, or default execution model for normal tool calls.
- The key architecture question is the boundary: what tmux may host versus what must remain owned by imp and mana.

Steps:
1. Ground the proposal in existing detached/background execution work and current CLI/runtime direction.
2. Define the allowed uses for tmux integration:
   - persistent interactive shell/session hosting
   - human attach/takeover/debugging
   - watch/log panes for long-running local work
3. Define the non-goals and guardrails:
   - do not treat tmux as durable task state or memory
   - do not make keystroke-sending pane automation the primary execution model
   - do not let hidden pane state become the canonical source of truth
4. Specify the boundary split:
   - tmux owns optional host-side terminal/session persistence and attachability
   - imp owns runtime behavior, command/tool execution semantics, operator UX, and structured capture of relevant session state
   - mana owns durable task state, evidence, verification outcomes, and resumable handoff
5. Identify integration affordances worth supporting if this moves forward:
   - session host creation and naming
   - attach/takeover
   - pane output capture into inspectable evidence/log surfaces
   - mapping tmux-backed sessions to detached/background run identity
6. Explain when tmux is appropriate versus when plain subprocess execution should remain the default.

Relevant references:
- root mana units `50.25`, `50.26`, `248.17`
- docs/rebuild/imp-cli-affordance-sequence.md
- docs/rebuild/imp-detached-execution-cli.md once present
- docs/rebuild/imp-detached-execution-model.md once present

In scope:
- architecture boundary for optional tmux integration
- operator/debugging/process-hosting use cases
- relationship to detached/background local execution and durable mana state

Out of scope:
- implementing tmux integration now
- making tmux a required runtime dependency
- replacing structured tool/subprocess execution with pane keystroke automation

Do not:
- present tmux as imp memory or orchestration
- rely on tmux pane history as the sole evidence trail
- broaden this into a generic terminal-emulator integration spec

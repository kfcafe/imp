## Task
Implement the runtime-side turn-scoped mana review packet described in `docs/rebuild/mana-between-turn-review-affordance.md`.

## Goal
Shift from per-call, last-write-wins mana delta widget updates to a turn-level aggregated review packet that another UI surface can render after each conversational turn.

## Grounded context
- `docs/rebuild/mana-between-turn-review-affordance.md` is the design source of truth.
- `imp/crates/imp-core/src/tools/mana.rs` currently emits per-action `set_mana_delta_widget(...)` updates such as created/updated/notes/decision-added/decision-resolved.
- `imp/crates/imp-core/src/agent.rs` owns turn lifecycle via `AgentEvent::TurnStart` / `TurnEnd` and is the likely place to begin/end a turn-scoped accumulator.
- `imp/crates/imp-core/src/tools/mod.rs` defines `ToolContext`, `ToolOutput`, and shared runtime-facing tool plumbing.
- `imp/crates/imp-core/src/ui.rs` already provides `set_widget`/`set_status` abstractions, but this unit should focus on producing the aggregated packet, not final rendering policy.

## Required behavior
1. Introduce a runtime-side aggregated mana review representation for one conversational turn.
2. Support explicit states equivalent to `no_change`, `changed`, and `needs_decision`.
3. Capture enough structured data to represent:
   - touched unit ids and titles
   - reused/created epic(s) when known
   - new child jobs
   - mutation kinds (notes appended, decision added/resolved, status/dependency changes)
   - unresolved decisions / consequential next question when available
4. Stop relying on only the last mana widget update as the durable between-turn story.
5. Keep the representation UI-neutral so TUI/CLI/RPC/Wizard can all consume it later.

## Implementation approach
- Add new review packet / accumulator types in an imp-core location that fits turn lifecycle ownership.
- Thread an accumulator through the runtime so mana mutations during a turn append structured records rather than only mutating a widget.
- Reuse existing structured `ToolOutput.details` from mana actions where possible instead of inventing a second ad hoc parser.
- Preserve current behavior enough that existing per-action widget/status updates do not break unrelated run-state UX in this unit.
- Prefer adding a small explicit helper/API for mana mutation recording over burying JSON-shape assumptions across the agent loop.

## Files
- `docs/rebuild/mana-between-turn-review-affordance.md` (read for contract)
- `imp/crates/imp-core/src/agent.rs` (modify — turn lifecycle integration)
- `imp/crates/imp-core/src/tools/mod.rs` (modify if ToolContext/runtime plumbing needs a review accumulator handle)
- `imp/crates/imp-core/src/tools/mana.rs` (modify — emit structured mutation records into the accumulator)
- `imp/crates/imp-core/src/ui.rs` (read only unless a small shared type belongs there)

## Acceptance criteria
- `imp-core` contains explicit turn-level mana review types or equivalent runtime abstraction.
- The code clearly distinguishes no-change vs changed vs needs-decision.
- Mana tool mutations can accumulate into the turn-level structure instead of only overwriting a widget string.
- Final rendering decisions remain separable from the core packet/accumulator.

## Do not
- Do not require Wizard.
- Do not implement final TUI visual polish here.
- Do not hard-code this feature only for the current TUI.
- Do not replace unrelated orchestration/run-status widgets with this turn-level packet.

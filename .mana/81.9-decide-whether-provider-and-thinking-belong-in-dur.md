Goal: decide whether per-unit `provider` and `thinking` should become durable mana-backed worker metadata, beyond the existing `model` field.

Current repo state inspected in this session:
- `mana-core::unit::Unit` currently exposes `model: Option<String>` but no durable `provider` or `thinking` fields.
- `tower-contracts::worker::WorkerAssignment` currently carries only `model: Option<String>` for selection-related durable metadata.
- `imp-cli` / `ImpSession::create()` already support transient runtime overrides for `model`, `provider`, and `thinking`.
- Planned native `imp` delegation work under 81.5/81.6 can improve transient override support without changing durable schema.

Question to answer:
Should durable unit metadata remain model-only, with provider/thinking chosen transiently at execution/delegation time, or should `provider` and/or `thinking` become first-class durable fields on units and worker assignments?

Steps:
1. Inspect current ownership expectations across `mana-core::Unit`, `tower-contracts::worker::WorkerAssignment`, and imp worker loading/runtime.
2. Compare the semantic role of each field:
   - `model` as durable unit preference/constraint
   - `provider` as possible execution routing hint or explicit contract
   - `thinking` as runtime posture versus durable task requirement
3. Decide which of these belong durably in mana versus transiently in imp runtime/tool invocation.
4. If durable fields are justified, specify exact contract changes and migration impact.
5. If not, record the reason and keep 81.5/81.6 scoped to transient overrides only.

Files:
- `mana/crates/mana-core/src/unit/mod.rs` (read/possibly later modify — current durable unit metadata)
- `tower-contracts/src/lib.rs` (read/possibly later modify — worker assignment contract)
- `imp/crates/imp-core/src/mana_worker.rs` (read — assignment loading/runtime seam)
- `imp/crates/imp-core/src/imp_session.rs` (read — transient model/provider/thinking resolution)
- `imp/crates/imp-cli/src/main.rs` (read — current `imp run` override behavior)

In scope:
- ownership decision for durable vs transient model/provider/thinking metadata
- explicit cross-project contract recommendation

Out of scope:
- implementing the transient override plumbing already scoped to 81.5/81.6
- broad runner protocol redesign

Do not:
- duplicate transient runtime knobs durably without a clear ownership reason
- assume provider/thinking should be durable just because CLI supports them transiently

Verify: create a durable root-scope record of the decision or design note, or at minimum a worker-ready contract unit describing the chosen ownership boundary.

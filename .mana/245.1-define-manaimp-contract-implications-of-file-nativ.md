## Task
Write a root architecture/contract note that translates the file-native memory model into mana↔imp contract implications. The goal is not full implementation but a precise decomposition another worker can execute cold.

## Current design direction
The memory architecture being adopted is:
- canonical truth = immutable durable objects
- materialized state = derived/rebuildable views
- runtime context = ephemeral and non-authoritative
- durable truth should be expressed through scoped claims backed by evidence and provenance

## What to specify
1. The durable object families that the contract boundary must carry:
   - observation objects: `attempt`, `checkpoint`, `evidence`, `receipt`, `env`
   - judgment objects: `claim`, `decision`, `lesson`, `runbook`
2. The metadata that must be available across the boundary:
   - object id/kind
   - scope fields such as project/task/path/commit/env where relevant
   - provenance fields such as producer, timestamps, cited evidence, supersedes/contradicts links
3. Derived-vs-canonical rules:
   - accepted facts/current state/open questions/conflict maps/stale queues/bootstrap packets are derived views
   - views may be cached or materialized but must never be the sole source of truth
4. Retrieval/compaction responsibilities:
   - checkpoints are handoff artifacts
   - attempt streams are raw history
   - role/task-specific context should be assembled from bootstrap + latest checkpoint + relevant lessons/runbooks/claims before dereferencing raw evidence
5. Parallelism/retry implications:
   - retries create new attempts with lineage
   - receipts prevent duplicate side effects
   - workers append objects rather than co-editing shared truth files

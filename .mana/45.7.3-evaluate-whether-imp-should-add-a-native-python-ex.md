Goal: capture the current architectural assessment and next-step decision criteria for a possible imp-native Python execution surface.

Current assessment from repo-grounded review:
- imp currently treats `bash` as the general execution backbone; the bash tool explicitly covers `python` usage and already has timeout/workdir/cancellation/truncation behavior.
- Current tool-shaping direction favors fewer native tools unless they provide clear structure or policy beyond shell execution.
- imp already has a first-class hosted extension story through Lua, so a Python surface must avoid becoming a second muddy extension/runtime path.
- A native Python tool is not currently necessary if the need is only occasional convenience for one-off snippets.
- A native Python tool becomes compelling only if it is a controlled compute substrate with stronger policy, provenance, artifact capture, and structured I/O than bash.

Questions this unit should answer:
1. Should imp continue to use bash for all Python execution for now?
2. If a native Python surface is later added, should it be framed as restricted local compute rather than general scripting or extensibility?
3. What contract would make a native Python tool materially better than bash instead of redundant with it?
4. How should it relate to Lua extensions and mode/capability policy?

Expected output:
- A concise architecture note or design section that records:
  - current recommendation (`not necessary yet` unless policy/provenance needs justify it)
  - criteria that would trigger adding a native Python tool later
  - a minimal proposed contract for such a tool if pursued
  - explicit non-goal: do not add a generic arbitrary-Python runner that merely duplicates bash

Likely design conclusions to preserve:
- native Python is valuable only if it gives structured inputs/outputs, tighter runtime policy, clearer provenance, and explicit artifact handling
- native Python is not justified purely for snippet convenience or nicer heredocs
- if added, it should likely be a narrow `python_run`/`python_exec`-style compute surface with explicit fields like `code|script_path`, `args`, `workdir`, `timeout`, `output_format`, interpreter selection, and artifact capture
- if the real goal is extensibility, strengthen Lua rather than introducing Python as a parallel extension substrate

Scope:
- in scope: architectural evaluation, capability boundary, interaction with agent modes/policy, minimal contract sketch
- out of scope: immediate implementation of a Python tool, package-management sandboxing claims that are not actually enforced, or replacing bash as the general executor

  Initial diagnosis from code inspection:
  - imp spawn ad_hoc path blocks on ImpSession::prompt_and_wait -> session.wait() -> joining the full agent task. If the provider/tool loop never emits AgentEnd because a downstream tool never resolves, spawn appears to run forever from the caller's perspective.
  - imp unit spawn path blocks on mana_worker::run_worker_assignment. After agent completion it runs inline verify via run_verify_command with TokioCommand::output() and no timeout, then close_unit_after_verify. Any hanging verify command will hang the spawn tool forever.
  - mana tool background run path itself is intentionally non-blocking when background=true, but foreground/native evaluate paths depend on run_state transitions driven by spawn_blocking(run_with_stream_capture_and_sink). If mana run never returns, run_state stays starting/running indefinitely.
  - mana-core worker close path uses api::close_unit -> mana_core::ops::close. Need deeper inspection there, but the more immediate hard hang risk already exists before close in worker inline verify.
  - mana-cli MCP close handler also runs verify via std::process::Command::output() with no timeout, so tool-driven close can block forever on a stuck verify command.
  - legacy mana spawner itself is non-blocking, but spawned worker templates can hang indefinitely if the child process never exits; runner timeout behavior in direct-mode needs a deeper check if user reports involve mana run/spawn orchestration.

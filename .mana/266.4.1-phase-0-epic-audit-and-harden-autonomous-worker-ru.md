Goal: make imp's autonomous execution paths trustworthy enough to use as the default way to run bounded work.
Current state: the intended split is documented (`imp run <unit-id>` = canonical single-unit worker runtime; `mana run` = orchestration / legacy compatibility dispatch), but user confidence is low and repo comments/docs still describe some paths as transitional. We need an explicit audit and proving plan before broader autonomous-tool investment.

Steps:
1. Audit the current `imp run <id>` path end-to-end: assignment loading, prompt/context assembly, provider/auth resolution, finalization, verify/close behavior, and user-visible failure modes.
2. Audit the current `mana run` path end-to-end: direct mode, template mode, background/native run state, spawn/log capture, and interactions with `imp run` workers.
3. Identify mismatches between intended architecture and current shipped behavior, especially where docs say canonical but code is still compatibility-oriented.
4. Define a compact proving/eval matrix for autonomous execution flows so future fixes can be verified repeatedly instead of by anecdote.
5. Sequence the first hardening changes with the smallest high-confidence wins first.

Files and areas likely relevant:
- imp/crates/imp-cli/src/lib.rs (run command wiring)
- imp/crates/imp-core/src/mana_worker.rs (canonical single-unit worker runtime)
- imp/crates/imp-core/src/tools/mana.rs (native mana tool orchestration and run-state)
- mana/crates/mana-cli/src/commands/run/mod.rs (legacy/compat orchestration path)
- mana/crates/mana-cli/src/spawner.rs (compat process spawn/log capture)
- docs/proposals/imp-run-worker-contract-and-mana-run-handoff.md
- docs/rebuild/imp-delegation-tool-and-runtime.md
- docs/rebuild/mana-imp-ownership-boundary.md
- imp/vNext.md

In scope:
- planning, audit, gap mapping, and proving strategy for autonomous runtime paths
- identifying concrete child jobs for code hardening

Out of scope:
- broad CLI/TUI polish
- unrelated provider/model feature additions
- rewriting the mana/imp boundary from scratch

Do not:
- assume docs equal reality without inspecting code
- leave worker-facing unit descriptions vague
- broaden into general UX cleanup before runtime confidence improves

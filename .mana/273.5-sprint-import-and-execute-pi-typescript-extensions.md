Goal: one sprint to make Pi TypeScript extensions both importable and executable in imp.

Current state:
- Pi import/detection/copying exists in working tree for `.imp/extensions/pi/<name>`.
- Bun-backed TS runtime execution is proven with Bun 1.3.13.
- Proven compatibility fixtures now include synthetic `registerTool`, local Pi `color-palette`, and reduced official Pi `dynamic-tools`.
- Bun bridge supports Pi-style `registerTool`, TypeBox-ish schemas, `session_start` lifecycle dynamic registration, basic UI/session stubs, and text/details tool callback execution through imp's Rust `ToolRegistry`.
- Remaining sprint risk is maintainability/support-boundary clarity before adding more compatibility.

Storage decision:
- Project/local imp extension root: `.imp/extensions/<source>/<extension-name>/`.
- Imported Pi extensions: `.imp/extensions/pi/<name>/...`.
- Use language-neutral `extensions` naming rather than `ts`/`typescript`.

Next execution order:
1. Refactor the spike into named seams without behavior changes:
   - generic TS extension loader/discovery/registry integration
   - Bun runner/bridge execution
   - Pi compatibility prelude/API shim/import rewrites
   - TypeBox/JSON schema normalization
   - fixture tests
2. Define and surface the sprint-1 support boundary.
3. Add user-visible validation/status so import/startup can distinguish executable, stubbed, and unsupported extensions.
4. Test one moderate dependency-bearing extension after the seams are clean.
5. Plan viewer/rendering support by first preserving renderer metadata and reporting custom renderers as unmapped.

Acceptance:
- `imp import --from pi` imports TS extensions into `.imp/extensions/pi`.
- imp can discover and execute at least one Pi-style TS extension tool through the normal tool registry using Bun.
- At least one selected compatibility fixture is non-redundant or externally sourced.
- Existing Lua extension behavior remains working.
- Limitations are documented honestly.

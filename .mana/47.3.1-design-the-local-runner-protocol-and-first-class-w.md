Goal: define the first local runner boundary for imp. Current state: execution is still shaped heavily by runtime-local helpers and shell-driven orchestration. We need a typed local runner MVP before implementation starts.

In scope:
- define the minimal runner request/response protocol
- decide what metadata a worktree lease needs
- decide what metadata a sandbox spec needs
- define how execution artifacts bind to runs, worktrees, and sandbox instances

Out of scope:
- full implementation of remote/daemon runners
- broad CI/CD or deployment orchestration beyond local execution boundaries

Important decisions to resolve explicitly:
1. should the first runner crate live inside imp or as a Tower sibling layer?
2. what exactly is in scope for v1 local runner vs deferred?
3. which worktree and sandbox fields must be durable in mana vs ephemeral in the runner?
4. how much PTY/session behavior belongs in the runner boundary from day one?

Suggested follow-ups:
- introduce runner request/result types
- wrap command execution behind the protocol
- add worktree lease records
- add sandbox profile definitions for edit/verify/reviewer/hostile cases

Replace the current pragmatic panic-hook-plus-CLI-fallback approach for imp interactive mode with a cleaner guarded runtime boundary. The goal is to centralize terminal teardown for returned errors and top-level panics inside the interactive runner/session boundary, so fatal output always appears after normal terminal restoration without relying primarily on global panic-hook state.

Current state:
- `imp-tui/src/terminal.rs` installs a scoped panic hook that restores the terminal before delegating to the previous panic hook.
- `imp-cli/src/lib.rs` calls `restore_process_terminal()` after `run_interactive()` errors as a belt-and-suspenders fallback.
- This fixes the user-visible bug but spreads fatal-path responsibility across terminal/session and CLI layers.

Steps:
1. Inspect `imp-tui/src/interactive.rs`, `imp-tui/src/terminal.rs`, and `imp-cli/src/lib.rs` to choose the narrowest boundary that fully owns interactive enter/run/exit semantics.
2. Introduce a guarded interactive execution path that catches top-level panics around the interactive run loop, restores the terminal exactly once, and returns a structured error for CLI printing.
3. Move non-panic fatal-path teardown responsibility out of `imp-cli` and into the guarded interactive boundary if possible.
4. Re-evaluate whether the scoped panic hook is still needed. Keep it only for truly uncaught/background panics that bypass the guarded runner, or remove it if the runner boundary fully covers the relevant failure modes.
5. Add focused tests proving: (a) top-level fatal errors restore before printing, (b) panic path restores before printing, and (c) restoration ownership is not duplicated across layers.

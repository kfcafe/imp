Goal: make streamed errors in imp both more user-friendly and more robust across human-facing and machine-facing surfaces.

Current state observed in this session:
- imp-core forwards provider StreamEvent::Error and stream Err cases into AgentEvent::Error, but the payload is just a raw string.
- imp-core also treats silent EOF without MessageEnd as an error, but surfaces a technical string rather than a presentation-oriented summary.
- imp-cli human/headless printing currently writes raw `Error: {error}` lines and only truncates tool errors; it does not share the TUI's API-error cleanup.
- imp-tui has a local `parse_api_error()` helper that extracts JSON provider messages and collapses HTML error pages, but this logic is duplicated only there and not reused by CLI/RPC.
- JSON/RPC output currently emits flat `{type:error,error}` / `{type:stream_error,error}` strings rather than a richer stable envelope.

Likely implementation seams:
- imp/crates/imp-llm/src/stream.rs
- imp/crates/imp-core/src/agent.rs
- imp/crates/imp-core/src/imp_session.rs
- imp/crates/imp-cli/src/lib.rs
- imp/crates/imp-tui/src/app.rs
- possibly provider parsers in imp/crates/imp-llm/src/providers/* and retry logic in imp/crates/imp-core/src/retry.rs

Desired outcomes:
1. Human-facing CLI/TUI errors are concise, actionable, and avoid dumping raw HTML/JSON blobs.
2. Partial-stream failures and silent EOFs are clearly distinguished from normal completion.
3. Shared formatting/parsing lives in core/shared code instead of only TUI.
4. Machine-facing JSON/RPC output can preserve raw details while adding a stable structured error shape.
5. Retry/startup-stream edge cases remain hardened and tested.

Open scope choice to resolve with user:
- first slice may target only human-facing CLI/TUI output, or also include JSON/RPC/stable machine envelopes in the same pass.

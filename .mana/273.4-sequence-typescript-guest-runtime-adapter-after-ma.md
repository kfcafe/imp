Goal: define the dependency-gated implementation path for executable TypeScript extensions in imp after import/staging support lands.

Updated sequencing based on local audit and existing imp import command:
1. First extend `imp import --from pi` to stage Pi TypeScript extensions under imp config and mark them unsupported-runtime/not-executable.
2. Then add host-neutral extension metadata/loading concepts in imp so staged packages are visible to status/help surfaces without being executed.
3. Then define host capability grants for TypeScript extension APIs equivalent to the Pi API surfaces imp chooses to support: tool registration, command registration, lifecycle hooks, UI interactions, native tool calls, env/secrets, process, network, filesystem.
4. Then select and implement a TS guest runtime adapter behind the Rust host boundary.

Runtime adapter responsibilities:
- Load one or more staged entrypoints.
- Provide a Pi-compatible `ExtensionAPI` shim where feasible.
- Capture calls to `registerTool`, `registerCommand`, and `on` and map them into imp's host registry/hooks.
- Convert TypeBox schemas into provider-compatible JSON Schema accepted by imp's tool system.
- Normalize errors, cancellation, timeouts, and dependency/install failures.
- Respect capability policy before exposing process/network/filesystem/env/secrets/native tool access.

Engine decision criteria:
- sandboxability and host bridge safety
- ability to run existing `.ts` entrypoints or transpile/load them cheaply
- package dependency story for extensions with package-local deps
- distribution/install footprint
- startup latency

Out of scope until prerequisites land:
- Executing TypeScript extensions as part of `imp import`.
- Routing worker execution through TypeScript.

Acceptance:
- There is a clear, dependency-gated path for executable TypeScript extensions that does not block the import-first compatibility work.

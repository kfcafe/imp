Goal: strengthen `imp run` into the canonical worker/runtime for executing a single mana unit, while preserving `mana run` as the orchestrator that selects ready work, runs many workers in parallel, owns verify batching, and records durable run state.

Current state:
- `mana run` already works as a parallel orchestrator and direct mode now prefers/uses `imp` rather than `pi`.
- `imp run <unit-id>` in `imp/crates/imp-cli/src/main.rs` is still a relatively thin headless path: it finds `.mana/`, scans markdown files in `load_mana_unit()`, assembles prompt/context in `run_headless_mode()`, runs the agent, then shells out to `mana close` after verify.
- `imp` already has a richer first-class native mana runtime in `imp/crates/imp-core/src/tools/mana.rs` (`run`, `run_state`, `evaluate`, background run tracking, summaries, widget/status updates), so mana-aware UX is currently split across two execution paths.
- Archived unit `29.7` decided against adding a new native imp orchestration/subagent tool action. This work must preserve that decision: strengthen the existing `imp run` CLI/runtime contract rather than reintroducing orchestration inside imp.

Steps:
1. Specify the canonical worker contract for `imp run` when used by `mana run`, including explicit mana root resolution, machine-readable protocol, and verify/close handoff semantics.
2. Define the execution bundle / unit-loading boundary so imp no longer relies on ad hoc markdown filename scanning as its long-term worker substrate.
3. Extract or implement a reusable imp-side unit worker runtime in imp-core, with `imp-cli` as a thin wrapper around that runtime.
4. Update `mana run` to invoke the strengthened `imp run` worker contract in parallel while retaining orchestration ownership in mana.
5. Polish direct human `imp run <id>` UX and focused tests so the same runtime serves both machine and human entrypoints cleanly.

Files:
- imp/crates/imp-cli/src/main.rs (current `run_headless_mode()`, `load_mana_unit()`, `ManaUnit::task_prompt()`, `task_context()`)
- imp/crates/imp-core/src/tools/mana.rs (native mana runtime/state UX to align with)
- imp/crates/imp-core/src/system_prompt.rs (headless execution contract)
- mana/crates/mana-cli/src/commands/run/mod.rs
- mana/crates/mana-cli/src/commands/run/ready_queue.rs
- mana/crates/mana-core/src/api/mod.rs or other canonical mana loading surface if execution-bundle APIs land there

In scope:
- strengthening `imp run` as the single-unit worker runtime
- preserving `mana run` as the orchestrator and parallel dispatch surface
- machine-readable worker protocol / outcome contract
- canonical unit-loading and execution-bundle boundary
- verify/close handoff semantics between imp and mana

Out of scope:
- introducing a new native imp orchestration tool action
- moving dependency scheduling / orchestration policy into imp
- replacing `mana run` with direct imp parallelism

Do not:
- re-litigate 29.7 as though imp should become its own orchestrator
- keep markdown filename scraping as the intended long-term worker contract
- leave close/verify ownership split across shell side effects if a cleaner contract is available

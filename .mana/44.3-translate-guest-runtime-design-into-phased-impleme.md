Goal: decompose the guest-runtime redesign into concrete future implementation jobs after the design docs are written.
Current state: no implementation plan exists yet for how to move from `lua_tool_loader` and the current `extend` tool to a generalized guest-runtime substrate.

Steps:
1. After the two proposal docs exist, write a short implementation-plan doc or mana-authored breakdown that turns the architecture into concrete jobs.
2. Break the work into phases with dependencies, including at least:
   - host/runtime abstraction in `imp-core`
   - manifest + permission model
   - Lua migration/adaptation
   - `extend` evolution into extension authoring/management
   - optional script tool implementation spike after policy is approved
3. For each phase, name likely files, expected artifacts, and a targeted verify strategy.
4. Record sequencing decisions so later workers do not start with the script tool before host policy exists.

- `docs/proposals/guest-runtime-implementation-plan.md` (create — if a doc is the clearest form)

In scope:
- sequencing, decomposition, verify strategy, likely file surfaces

Out of scope:

Goal: Write a comprehensive implementation contract before coding the Dirac-inspired improvements to existing imp tools. Do not add new model-facing tools unless later explicitly approved; improve `read`, `scan`, and `edit`.

Current state:
- Dirac comparison in parent 280 and 280.2 found the key gaps: AST/symbol-native context, hash/anchor-backed edit protocol, batching, diagnostics, and eval reporting.
- User clarified: anchors should be part of `read`; no additional tools are needed. Focus on improving existing `read`, `scan`, and `edit` tools.
- User raised naming question: maybe `scan` should be `search` or `codesearch`. The spec must evaluate naming and compatibility. Bias: do not abruptly rename in v1; consider clearer descriptions/action names or a compatibility alias only if justified.
- imp currently has generic `scan`, `read`, `edit`, `multi_edit`, file tracking, and checkpoints.
- Local code evidence:
  - `crates/imp-core/src/tools/scan/mod.rs` already implements `ScanTool`, `extract_symbol`, `extract_blocks_at_lines`, parser selection, and formatted outlines.
  - `crates/imp-core/src/tools/read.rs` reads stable line-oriented output and records file reads.
  - `crates/imp-core/src/tools/edit.rs` supports exact/fuzzy `oldText` replacement and delegates `edits` arrays to `multi_edit`.
  - `crates/imp-core/src/tools/mod.rs` has `ToolContext` with `file_cache`, `checkpoint_state`, and `file_tracker`.

Spec requirements:
1. Create `docs/design/dirac-inspired-code-tools.md`.
2. Include a concise problem statement: imp already has durable runtime/mana/checkpoints; this spec closes Dirac's code-modification efficiency gap by improving the existing code tools.
3. Include a naming section for `scan` vs `search` vs `codesearch`:
   - Evaluate user/model clarity, collision risk, backwards compatibility, and actual semantics.
   - Recommend a v1 path. Likely: keep canonical `scan`, improve description/actions/output, maybe document `scan` as code-structure search; defer alias/rename until eval evidence.
4. Specify v1 changes to existing tools only:
   - `read`: optional anchors in line-oriented output; refreshed metadata after anchored edits; clear stale behavior.
   - `scan`: sharper actions/output for skeletons and symbol extraction, preferably by enhancing `scan`/`build`/`extract`; no new tool name in v1.
   - `edit`: accept anchor ranges or anchored replacement mode in addition to exact `oldText`; preserve current exact/fuzzy semantics; keep batch edits inside `edit`/`multi_edit`.
5. Specify exact proposed JSON schemas/parameters and output shapes for:
   - `read` with anchors disabled/enabled.
   - `scan` actions: `scan`, `build`, `extract`; optionally renamed/aliased action labels only if backward compatible.
   - `edit` exact mode, anchored mode, and `edits` array/multi-edit mode.
6. Specify shared result envelope fields where useful: path, language, symbol, start_line, end_line, anchors/file hash when applicable, stale status, diagnostics summary, truncation metadata, checkpoint/evidence references.
7. Specify invariants:
   - anchors are opaque, file-scoped, session-local, and not security tokens.
   - anchor edits require current read/anchor state.
   - destructive operations checkpoint first.
   - exact `edit` remains supported and is the compatibility path.
   - no IDE dependency in v1.
   - no broad new dependencies without approval.
8. Specify implementation sequencing with child units 280.2.2, 280.2.3, 280.2.4 and any additional follow-up specs needed.
9. Include edge cases:
   - unread file edit warnings.
   - stale anchors after external modification.
   - duplicate line content.
   - CRLF/newline handling.
   - binary/huge files.
   - partial reads/offset+limit with anchors.
   - overlapping multi-edits.
   - fuzzy edit fallback risk.
   - generated/vendor files.
10. Include test strategy and targeted verify commands.

- Replace existing `edit` semantics in the spec.
- Create separate `anchor_read`, `anchor_edit`, `symbol_extract`, or `code_skeleton` tools in v1.
- Rename `scan` without explicit migration plan and compatibility story.

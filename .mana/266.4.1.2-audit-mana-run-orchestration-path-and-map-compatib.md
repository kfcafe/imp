Goal: produce an evidence-based gap map for `mana run` as the orchestration/compatibility path into imp workers.
Current state: docs and code describe `mana run` as orchestration / legacy compatibility, with direct mode able to spawn `imp run <id>`, plus native/background run-state surfaces in imp's mana tool. User confidence is low, so we need a crisp picture of what actually works and what remains transitional.

Steps:
1. Inspect `mana/crates/mana-cli/src/commands/run/mod.rs` and document direct mode, template mode, loop/background behavior, ready-queue dispatch, and how logs/run-state are produced.
2. Inspect `mana/crates/mana-cli/src/spawner.rs` and summarize compatibility-process spawning, log capture, and failure/exit handling.
3. Inspect `imp/crates/imp-core/src/tools/mana.rs` only for the native in-session run-state/log surfaces that overlap with orchestration visibility.
4. Identify where current code still reflects shell-template/compat assumptions instead of the intended long-term `imp run` worker boundary.
5. Propose the smallest hardening tasks or deprecations that would make orchestration behavior understandable and trustworthy.

Files:
- mana/crates/mana-cli/src/commands/run/mod.rs (read — orchestration path)
- mana/crates/mana-cli/src/spawner.rs (read — compat spawn/log capture)
- imp/crates/imp-core/src/tools/mana.rs (read — native mana tool run-state/log view)
- docs/rebuild/imp-delegation-tool-and-runtime.md (read — intended split)
- docs/rebuild/mana-imp-ownership-boundary.md (read — ownership model if needed)

Context:
- `mana run` is not the desired long-term boundary language, but it still exists and needs to be reliable while present.
- The audit should distinguish: current shipped behavior, compatibility debt, and user-facing confusion.

In scope:
- audit and planning only
- concrete follow-up job proposals

Out of scope:
- broad mana architecture redesign
- changing code

Do not:
- assume the background/native run-state surfaces cover every legacy path
- blur orchestration concerns with single-worker runtime concerns

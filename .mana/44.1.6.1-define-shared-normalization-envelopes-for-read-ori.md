## Goal
Define the common runtime result envelopes and shared foundational types for Phase 1 read-oriented code intelligence in `imp`.

## Current state
The completed read-oriented planning doc recommends stabilizing what the model sees before multiplying adapters. Shared shapes are needed for locations, symbols, diagnostics, references, hover/signature cards, structural results, backend provenance, and freshness.

## Scope
- Planning only.
- Normalize result shapes exposed by `imp` runtime surfaces.
- Cover both AST-backed structural results and semantic query results.
- Include enough alignment with compiler/linter/build/test problem summaries for cross-surface reasoning.

## Acceptance
- A follow-on unit can implement common envelopes for `surface`, `backend`, `query`, `summary`, `items`, `truncated`, `warnings`, and suggested-next-query fields.
- Shared types are identified for `Location`, `SymbolIdentity`, `RelatedLocation`, and a problem-list item that can represent semantic and deterministic check findings without flattening them into one protocol.
- The unit description keeps raw backend payloads as internal/debug detail rather than the primary model-facing interface.

## Important constraints
- Do not overdesign a globally stable cross-backend identity system in the first slice.
- Do not assign any of this normalization ownership to `mana` runtime orchestration.
- Keep provenance and freshness explicit on every normalized result.

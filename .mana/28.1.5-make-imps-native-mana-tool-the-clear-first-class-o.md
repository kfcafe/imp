Goal: align imp's native mana tool UX with the strengthened imp worker contract so the intended first-class user path is clean, visible, and explicitly layered.

Current state:
- The native mana tool in `imp/crates/imp-core/src/tools/mana.rs` is already the intended first-class orchestration UX inside imp sessions: it supports `run`, `run_state`, `evaluate`, `logs`, background runs, summaries, and widget/status updates.
- However, the current 28.1 decomposition over-emphasized direct human-facing `imp run <id>` polish. The clarified product intent is that imp's native mana tool is the primary first-class UX, with `mana run` as orchestrator and `imp run` as worker runtime.
- Once 28.1.1–28.1.3 land, the native mana tool should clearly surface that stack rather than looking like a thin pass-through.

Steps:
1. Read 28.1 notes plus 28.1.1–28.1.3 outputs.
2. Update native mana tool messaging, summaries, widget/status content, and follow-up guidance so it clearly presents itself as the first-class orchestration surface inside imp.
3. Ensure background run start, run-state, evaluate, and logs flows describe the worker/runtime stack accurately: native mana tool → mana orchestration → imp workers.
4. Add focused tests or output assertions in `imp/crates/imp-core/src/tools/mana.rs` for the clarified UX wording where practical.
5. Coordinate with 28.1.4 so docs distinguish primary native-tool UX from secondary direct `imp run` usage.

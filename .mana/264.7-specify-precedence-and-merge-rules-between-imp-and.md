Goal: define the exact precedence, merge, and shadowing rules for canonical imp-native roots after the root-policy decision.

Current accepted direction:
- canonical global root: `~/.imp/`
- canonical project-local root: `<project>/.imp/`
- XDG / macOS Application Support roots are compatibility + migration inputs only
- project-local `.imp/` should override or extend global `~/.imp/` only for project-aware behavior/content
- personal/global-only artifacts remain global-only

Scope of this job:
1. Specify exact lookup/merge behavior for each shared surface:
   - `config.toml`
   - `soul.md`
   - `skills/`
   - `prompts/`
   - `tools/`
   - `lua/`
2. Specify whether each surface is:
   - replace/override
   - additive union
   - additive with name shadowing
   - ordered stack by ancestry
3. Define project ancestry behavior where relevant (for example nested cwd lookup for project `.imp/skills` and `.imp/soul.md`).
4. Specify the global-only surfaces and explicitly forbid project-local variants for them:
   - `sessions/`
   - `indexes/`
   - `auth.json`
   - `memory.md`
   - `user.md`
5. Produce a worker-ready contract so implementation can add one central path/lookup API without re-deciding semantics.

Files to inspect/reference:
- `imp/crates/imp-core/src/resources.rs`
- `imp/crates/imp-core/src/config.rs`
- `imp/crates/imp-lua/src/loader.rs`
- `imp/crates/imp-core/src/tools/extend.rs`
- `imp/README.md`

Done when:
- every canonical root surface has explicit precedence semantics
- the contract distinguishes override vs extension clearly enough to implement and test
- global-only surfaces are explicitly called out

Out of scope:
- implementation refactor
- migration mechanics beyond what is needed to state source-of-truth behavior

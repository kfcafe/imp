Goal: define the first implementation sequence for making imp's current proto-SDK usable and legible without inventing a second parallel abstraction too early.
Current state: the core capabilities exist in `imp-core`, but lack a curated public surface, documentation story, and likely some runtime/lifecycle cleanup.

Sequence:
1. Identify the smallest documentation/export cleanup needed to make the existing session API a real SDK seed.
2. Identify the minimum refactors needed to separate SDK-safe surface from CLI/TUI-specific orchestration.
3. Identify the first verifyable implementation slice (for example: public re-exports/docs/examples/tests around `ImpSession` and event consumption).
4. Record what should explicitly wait for later phases (TS extensions, package ecosystem, provider expansion, richer host UI).

Files/context:
- `imp/crates/imp-core/src/lib.rs`
- `imp/crates/imp-core/src/imp_session.rs`
- `imp/crates/imp-cli/src/lib.rs`
- `imp/README.md`

Deliverable:
- a compact ordered sequence for the first SDK implementation pass.

Do not:
- jump immediately to a new crate split unless justified
- turn the first slice into a giant refactor

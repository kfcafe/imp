This epic turns the rebuild into a migration plan rather than a rewrite. Goal: define the order of operations, temporary adapters, verification strategy, and rollback points needed to land the refactor safely.

Focus:
- crate/module migration order
- compatibility shims and re-exports
- verify strategy for each extraction slice
- rollback points and integration discipline

Important decisions to resolve explicitly:
1. which refactors must happen before others because of ownership or protocol constraints?
2. where do temporary adapters belong, and how long are they allowed to live?
3. what user-visible behavior must remain stable throughout the migration?
4. which slices are safe to land independently vs only behind a preparatory adapter layer?

Suggested child jobs:
- define migration order for contracts, imp-core decomposition, runner boundary, and evidence model
- define temporary re-export/shim policy
- write focused verify commands for each migration slice
- identify rollback points for the highest-risk refactors
- decide what parts of the TUI or CLI should intentionally be left untouched until runtime seams are cleaner

Done when:
- the rebuild can proceed in small landable slices without forcing a flag-day rewrite

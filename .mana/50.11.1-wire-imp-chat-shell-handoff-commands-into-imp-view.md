  Durable decomposition imported from `docs/rebuild/imp-view-submodes-and-entrypoints.md` so implementation can inherit the product decision without re-opening it.

  Viewer product-shape decision:
  - Prefer a family of focused viewer entrypoints over one viewer shell with hidden tabs.
  - Canonical surface: `imp view sessions`, `imp view tree`, `imp view logs`, `imp view checkpoints`.
  - `imp view` with no area should default to `sessions` rather than opening a generic dashboard.

  Code-grounded rationale to preserve during implementation:
  - `session_picker.rs` already models a search-first list/detail browser with cwd-aware ranking and Enter-to-open; this should ground `imp view sessions`.
  - `tree.rs` already models a specialized branch/tree inspector with filter modes (`all`, `current-branch`, `branch-points`, `no-tools`, `user-only`) and selection semantics; this should ground `imp view tree`.
  - `sidebar.rs` already models read-heavy inspection in stream and split list/detail forms; this should ground `imp view logs`.

  Implementation-facing shell mapping to keep explicit:
  - `:view` -> `imp view sessions`
  - `:view sessions` -> `imp view sessions`
  - `:view tree` -> `imp view tree`
  - `:view logs` -> `imp view logs`
  - `:view checkpoints` -> `imp view checkpoints`

  Context rules when launched from `imp chat`:
  - pass current cwd/project bias into sessions browsing
  - pass current session context into tree/logs/checkpoints browsing (`--current-session` now; later explicit `--session <id>` if needed)
  - if no current session exists, fail honestly and point to `imp view sessions`

  Selection-and-return contract to preserve:
  - sessions may return selected session id
  - tree may return selected node/branch target
  - checkpoints may return selected checkpoint id
  - logs usually return without a selection payload
  - after return, the shell should resume with a compact result line rather than silently morphing into the viewer

  Non-goals for follow-on implementation:
  - do not turn `imp view` into a general fullscreen authoring interface
  - do not hide viewer mode switches behind a vague `imp view` dashboard
  - do not move model/settings/login/secrets control into the viewer surface
  - shared viewer runtime under the hood is fine; hidden product-surface tabs are not the intended contract.

Implement the next narrow follow-up after the first concrete `imp view` slice: add explicit shell-to-view handoff commands inside `imp chat`. Current state already grounded in `imp/crates/imp-cli/src/main.rs`: `imp chat` exists as a transcript-oriented shell; `imp view sessions`, `imp view tree`, `imp view logs`, and `imp view checkpoints` now exist as plain-text focused viewer entrypoints. Goal: let the shell invoke these browse-heavy surfaces without turning the shell itself into a browser.

Scope:
1. Extend the `imp chat` shell command parser to support `:view <area>` (and slash compatibility) for at least: `sessions`, `tree`, `logs`, `checkpoints`.
2. Implement explicit shell-to-view handoff behavior. Prefer invoking the same viewer code path/entrypoint semantics rather than duplicating viewer logic inline in the shell.
3. Keep the handoff explicit and legible: do not silently morph the shell into a browser-like mode.
4. Keep the shell transcript-oriented and single-pane after this change.
5. If a viewer area is unknown, show a concise usage/help message.

Relevant files:
- `imp/crates/imp-cli/src/main.rs`
- any small extracted helper module under `imp-cli` if needed, but avoid a broad refactor in this slice

Out of scope:
- interactive filtering/search inside `imp view`
- richer viewer UI polish
- guided settings/personality/setup flows
- changing the current `imp tui` surface

- `cargo check -p imp-cli`

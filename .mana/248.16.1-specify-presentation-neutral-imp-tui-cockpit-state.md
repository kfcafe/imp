Goal: convert the racecar/cockpit redesign direction into a presentation-neutral state model and layout grammar that future imp-tui work can implement without guessing.

Current state grounded this session:
- app.rs renders top bar, optional widget tray, transcript, optional sidebar, composer, and overlays directly from runtime/UI state.
- status semantics exist, but identity/run-state/attention/detail are not yet modeled as distinct cockpit strata.
- docs/design/imp-premium-interaction-system.md and docs/design/imp-tui-cockpit-wireframes.md define product posture, but not yet a concrete state model tied to current imp-tui seams.

Need:
- define the canonical cockpit strata: identity strip, run-state strip, transcript/road, telemetry rail, composer/control stick, attention/review surface
- define the canonical state vocabulary and which stratum owns each piece of information
- map current app.rs/view responsibilities onto the target grammar so later implementation does not improvise ownership
- capture transitions between drive/inspect/review/control-room/browse conceptual modes without forcing a full rewrite first

Ground against:
- imp/crates/imp-tui/src/app.rs
- imp/crates/imp-tui/src/views/{top_bar.rs,status.rs,editor.rs,sidebar.rs,chat.rs}
- docs/design/imp-premium-interaction-system.md
- docs/design/imp-tui-cockpit-wireframes.md

Deliverable:
- durable design note or spec that another worker can use to implement the first cockpit proving slice.

Out of scope:
- code changes to the TUI
- theme/palette bikeshedding divorced from layout/state semantics

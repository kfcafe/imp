## Task
Make the central architectural decision for v1 imp-to-imp delegation: what can a parent imp actually delegate to a child imp?

## Repo grounding
- `imp run <unit-id>` already exists and is the canonical single-unit execution path.
- `imp-core::mana_worker` already loads `WorkerAssignment` and shapes task context.
- `mana-cli` still owns compatibility orchestration via `run_native()` and `spawner.rs`.

## Decision options to evaluate
1. mana-addressed units only
2. freeform/ad hoc prompt delegation
3. two-layer model: transient ad hoc child calls allowed, but durable or mutating delegation must route through mana and `imp run`

## Output
Write the chosen v1 target model into the parent design note and explain:
- why it matches current repo reality
- what is explicitly allowed in v1
- what is deferred
- what durability guarantees the parent and child do or do not get

## Acceptance
- Recommendation is explicit, not left ambiguous.
- The note states whether direct prompt-to-imp delegation is allowed in v1.
- The choice is consistent with `mana → imp → mana` and current `imp run` primacy.

## Verify
Parent design note clearly names one of: mana-only, ad hoc, or two-layer as the v1 target model.

Goal: Improve imp's existing `scan` tool with Dirac-inspired read-oriented code intelligence. Do not add new model-facing tools unless later explicitly approved.

Current state:
- User clarified the desired direction: improve `read`, `scan`, and `edit`; avoid separate new tools.
- `crates/imp-core/src/tools/scan/mod.rs` already supports `scan`, `build`, and `extract` actions.
- It has internal helpers: `extract_symbol(source, path, name)`, `format_result`, `format_function`, `format_type`, and parser selection.
- Existing `scan extract` can target `file#symbol`, but the schema/output can be made more discoverable and useful for low-token refactors.

Steps:
1. After 280.2.1 lands, follow its contract for naming/schema/output.
2. Enhance the existing `scan` tool rather than adding `symbol_extract`/`code_skeleton` tools. Candidate improvements:
   - document/promote `extract` targets like `file#symbol`, `file:start-end`, and `file:line` in the schema description.
   - ensure `build` or `scan` output can serve as a concise file skeleton with function/type signatures and line ranges.
   - include structured details JSON where compatible: path, symbol, kind, language, start_line, end_line, truncation.
3. Reuse existing scan helpers instead of duplicating tree-sitter traversal.
4. Keep output compact and model-actionable.
5. Add targeted tests using existing or new small fixtures for Rust and TypeScript if the implementation changes parsing/output behavior.

Files:
- `crates/imp-core/src/tools/scan/mod.rs` (modify)
- `crates/imp-core/src/tools/mod.rs` (modify only if schema/registration metadata lives there)
- `crates/imp-core/tests` or in-module tests (modify/create as needed)

In scope:
- Existing `scan` tool improvements for skeleton/orientation and symbol extraction.
- Rust and TypeScript proving fixtures if behavior changes.

Out of scope:

- References, rename, replace, anchor edits, diagnostics.
- Full LSP integration.

  Superbar field grammar to plan against:
  - top-left edge: stable identity only, preferring home-abbreviated path plus optional session label (for example `~/imp · session-name`)
  - top-right edge: slow-changing operational context such as model, elapsed timer, and context pressure
  - bottom-right edge: active run-state/liveness label such as `thinking`, `working · 2 tools`, `streaming`, `queued`, `blocked`, or `review`
  - bottom-left edge: one short transient cue only, such as active tool focus, subagent count, queue count, or selected-object summary
  - inside composer: prompt/input text only; do not clutter the input region with status chrome

  Ownership split to preserve in the refactor plan:
  - transcript remains the compact road and should show terse tool rows only
  - superbar carries routine identity/status/queue/scan signaling
  - sidebar/inspector carries rich selected-tool detail plus mana/trust/review/blocked inspection

  Anti-drift rule:
  - if the superbar feels crowded, first shorten labels, compress metadata, or escalate details into the sidebar; do not solve normal crowding by adding a permanent second horizontal band

  Draft width-collapse priority to evaluate in the plan and wireframes:
  1. input always survives
  2. active run state survives
  3. path survives
  4. model survives
  5. timer survives
  6. context percentage survives
  7. session title drops before the above if needed
  8. subagent/tool-count extras drop earliest

  This collapse order is a proposed planning default, not yet final product law.

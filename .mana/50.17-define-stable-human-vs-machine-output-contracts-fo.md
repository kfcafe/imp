Write `docs/rebuild/imp-output-contracts.md` defining the stable output contract split required by the CLI-first sequence captured in `docs/rebuild/imp-cli-affordance-sequence.md`.

Ground the work in current repo/design material:
- `imp/IMP_REVIEW.md`
- `docs/rebuild/imp-cli-affordance-sequence.md`
- `docs/rebuild/imp-shared-runtime-startup-map.md`
- `docs/rebuild/imp-cli-shell-transcript-ux.md`
- `imp/crates/imp-cli/src/main.rs` output helpers such as `determine_headless_output_mode(...)`, `print_headless_human_event(...)`, `print_json_event(...)`, and the RPC JSON conversion path

The doc should:
1. define the stable distinction between human terminal output and machine-readable output for CLI-first `imp`
2. cover at least `imp prompt`, `imp run`, CLI shell-adjacent run output, and RPC / protocol output
3. identify which event categories/fields should be shared across surfaces versus which rendering details are presentation-specific
4. call out current duplication and say what must move behind a shared event/output seam rather than remain CLI-only formatting
5. explicitly explain why this affordance comes before the later CLI-native items in the sequence

Acceptance criteria:
- the document names both human and machine contracts explicitly
- it uses current code and docs as grounding rather than proposing a greenfield protocol
- it distinguishes shared-runtime contract work from terminal-presentation polish
- it does not implement the contract here; it only defines it clearly enough for later execution

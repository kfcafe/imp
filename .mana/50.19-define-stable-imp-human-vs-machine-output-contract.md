Write a rebuild-spec doc that turns the output-mode prerequisite from `docs/rebuild/imp-cli-affordance-sequence.md` into an explicit execution contract. Ground it in current repo/doc reality: `docs/rebuild/imp-shared-runtime-startup-map.md` shows output/event duplication across print, headless, RPC, and TUI; `docs/rebuild/imp-command-grammar.md` fixes the CLI-first surface families; `docs/rebuild/imp-cli-shell-transcript-ux.md` defines the desired human transcript rhythm. The doc should decide the minimum stable shared event/result categories that all surfaces consume, and then define two output adapters at the product surface: (1) human-readable transcript mode for `imp chat` and human-facing `imp run`, and (2) machine-readable structured mode for scripts, wrappers, and future detached/reattach inspection. Cover naming, guarantees, what belongs in shared runtime events vs adapter-specific envelopes, and why this must land before detached/background execution and serious approval pause handling. Keep scope architectural/spec-level, not implementation.

Acceptance criteria:
- explicitly contrasts current fragmented output handling with the target shared-event + adapters model
- names stable human vs machine output modes and their intended consumers
- states which later affordances depend on this contract (detached/background, approval-required state, structured reattach/status)
- stays repo-grounded in the inspected startup/output docs rather than inventing a greenfield protocol

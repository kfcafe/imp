This epic makes runtime capability boundaries real, consistent, and auditable. Goal: ensure tool visibility, execution permissions, environment access, shell access, network access, and guest-runtime behavior all line up with the intended mode/policy model.

Primary source concerns:
- Lua/guest runtime can bypass core policy boundaries
- cancellation does not propagate cleanly into active tool execution
- some mode/tool contracts do not perfectly align with actual runtime wiring
- shell/backend and env behaviors need clearer ownership

Important decisions to resolve explicitly:
1. should Lua remain the primary extension model or become one guest-runtime backend under a stricter abstraction?
2. what capabilities are deny-by-default for extensions?
3. what should the deterministic runtime policy interface look like?
4. what policies belong in prompt guidance vs execution enforcement?

Suggested child jobs:
- audit visible vs actually executable capabilities per mode
- design deterministic runtime policy hooks for tool and mana actions
- tighten guest runtime access to native tools, env, shell, and network
- propagate cancellation through active tool execution
- align shell/backend behavior with actual runtime configuration ownership

Done when:
- the capability hardening track is reduced to concrete implementation jobs instead of a broad safety concern bucket

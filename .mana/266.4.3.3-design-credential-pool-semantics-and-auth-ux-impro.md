Goal: define a minimal credential-pool and auth UX model for imp that improves resilience without importing Hermes complexity wholesale.

Current state:
- imp has `AuthStore` with runtime overrides, stored credentials, env resolution, and secure field storage.
- Hermes has a much richer credential pool (`agent/credential_pool.py`) with rotation, exhaustion cooldowns, and multiple entries per provider.

Steps:
1. Inspect imp auth surfaces:
   - `/Users/asher/imp/crates/imp-llm/src/auth.rs`
   - `/Users/asher/imp/crates/imp-cli/src/lib.rs` login/secrets flows
2. Compare against the reviewed Hermes pool semantics and extract only what fits imp.
3. Specify:
   - whether imp should support multiple credentials per provider
   - minimal exhaustion/cooldown semantics
   - operator-facing UX implications in CLI/TUI
   - what remains explicitly out of scope for the first slice
4. Keep the recommendation incremental and Rust-native.

In scope:
- credential-pool semantics
- auth UX improvements
- incremental implementation path

Out of scope:
- implementing full multi-provider orchestration in this unit

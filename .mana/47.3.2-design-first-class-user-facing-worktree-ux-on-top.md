Goal: define the human-facing worktree UX that makes imp feel more productized without bypassing the runner/worktree lease substrate in 47.3.

Current state:
- 47.3 already owns the runner/worktree/sandbox substrate.
- We now have explicit product direction from the opencode audit to make worktrees a visible user primitive, not just an internal implementation detail.

Steps:
1. Inspect current imp + mana execution entrypoints and identify where a human should be able to see, request, reuse, or clean up worktrees.
2. Define the minimum user-facing worktree model: when a unit gets a worktree, how it is named, how it is surfaced in CLI/TUI, how reuse vs fresh isolation works, and what cleanup path exists.
3. Specify how this UX maps onto the lease/runtime contract from 47.3 so UI shortcuts do not create a second shadow worktree system.
4. Capture recommended follow-on implementation slices for CLI/TUI/runtime.

Files/areas:
- imp/crates/imp-cli/src/main.rs
- imp/crates/imp-tui/src/app.rs
- imp/crates/imp-core/src/tools/mana.rs
- imp/crates/imp-core/src/system_prompt.rs
- mana runtime/worktree lease surfaces that 47.3 references

In scope:
- user-facing worktree UX model
- lifecycle states, naming, visibility, cleanup expectations
- mapping product UX to runner/worktree lease substrate

Out of scope:
- implementing the runner itself
- adding a separate worktree mechanism outside the 47.3 boundary

Do not:
- propose a session-centric design that weakens mana/unit grounding
- treat worktrees as purely hidden runtime details if the product goal is to make them legible

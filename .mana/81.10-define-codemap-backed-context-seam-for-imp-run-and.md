Goal: define the first explicit codemap/context-assembly seam that can enrich `imp run <unit-id>` and the future native `imp` delegation tool without blocking the current 81.5 -> 81.8 MVP.

Current state:
- `imp run` already assembles mana-backed task context through `imp_core::mana_worker::{load_assignment_with_mana_dir, build_task_context, build_task_prompt, assemble_prefill}` and task/session facts through `imp_core::mana_prompt_context`.
- Current context enrichment is file-prefill only (`imp/crates/imp-core/src/context_prefill.rs`), based on explicit file/path references plus simple `:tail:N` and `:start-end` suffixes.
- There is no first-class runtime surface named codemap/code map in the active imp runtime today.
- The native `imp` tool is not yet implemented; 81.5 -> 81.8 already tracks the MVP extraction, tool implementation, ad-hoc delegation mode, and policy gates.
- Broader read-oriented code-intelligence planning already exists in root mana under 44.* and should be reused rather than duplicated.

Steps:
1. Audit the current `imp run` context path in `imp-cli` and `imp-core` and write down exactly where mana facts, task context, and file-prefill are assembled today.
2. Distinguish three layers clearly: (a) mana-backed task/session prompt context, (b) current file-prefill/context cache, and (c) future codemap/semantic context. Be explicit about what is real now versus aspirational.
3. Define the narrowest first codemap seam that could be consumed by both `imp run` and the future native `imp` tool. Name the boundary, likely inputs/outputs, cacheability expectations, and failure/fallback behavior.
4. Explain how this seam should relate to existing code-intelligence planning under 44.* so the native imp-tool path does not fork a second semantic-stack design.
5. State whether codemap integration should remain post-MVP relative to 81.5 -> 81.8 or whether any minimal codemap artifact is worth pulling into the MVP; justify the answer from current repo reality.

Files:
- imp/crates/imp-cli/src/main.rs (read — current `imp run` headless path)
- imp/crates/imp-core/src/mana_worker.rs (read — canonical worker loading/task-context/prefill helpers)
- imp/crates/imp-core/src/mana_prompt_context.rs (read — mana-backed session/task prompt facts)
- imp/crates/imp-core/src/context_prefill.rs (read — current file-prefill implementation)
- docs/rebuild/imp-delegation-tool-and-runtime.md (read — current native delegation design baseline)
- docs/rebuild/imp-run-primary-execution-path.md (read — current imp-run primacy audit)
- root mana units 44.*, especially read-oriented code-intelligence planning (read — avoid duplicating broader semantic-stack design)
- docs/rebuild/imp-codemap-context-seam.md (create)

In scope:
- Narrow architecture/spec for codemap-backed context assembly as consumed by `imp run` and native delegation.
- Explicit current-state audit of mana prompt context vs file-prefill vs future codemap.
- MVP sequencing recommendation relative to 81.5 -> 81.8.

Out of scope:
- Implementing semantic/LSP-backed codemap runtime.
- Redesigning the entire code-intelligence roadmap already tracked under 44.*.
- Changing durable mana schema beyond what the seam description requires.

Do not:
- Claim codemap support already exists if the repo only has file-prefill.
- Duplicate the 44.* code-intelligence plan instead of linking/aliging with it.
- Turn codemap into an implicit blocker for the native imp-tool MVP without explicit evidence.

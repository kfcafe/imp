Goal: polish the direct human `imp run <id>` UX and verify/document the unified worker runtime once the shared worker contract exists.

Current state:
- Today the human-facing `imp run <id>` experience is effectively the same thin headless machinery that mana dispatch uses.
- After 28.1.2 and 28.1.3, the same stronger worker runtime should serve both machine and human entrypoints, but the human UX still needs explicit polish and documentation.

Steps:
1. Read the architecture spec from 28.1.1 and the resulting runtime behavior from 28.1.2/28.1.3.
2. Improve direct `imp run <id>` messaging so unit assignment, progress, blocker reporting, verify/defer mode, and final outcome are clear to a human operator.
3. Add or update focused docs describing the split: `mana run` orchestrates many units in parallel; `imp run` executes one unit using the canonical worker runtime.
4. Add focused CLI/runtime tests for human-facing behavior where practical.

- TUI-wide redesign unrelated to `imp run`

- create a second separate runtime path for human mode if the shared worker runtime can serve both
- document the split in a way that suggests imp owns multi-unit orchestration
